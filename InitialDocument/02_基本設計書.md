## 1. アーキテクチャ設計

### 1.1 コンポーネント一覧

| コンポーネント | 層 | 役割 | 稼働形態 |
|---|---|---|---|
| market_collector | L1 | 価格・板のWS受信、ローソク足生成、DB保存(FR-1.1) | 常駐(systemd) |
| news_collector | L1 | RSS・指標カレンダーのポーリング、正規化(FR-1.2〜1.3) | 常駐 |
| news_pipeline | L1+L2 | 3段フィルタ。Stage1はルール、Stage2/3でClaude API呼び出し(FR-2) | 常駐(イベント駆動) |
| council_runner | L2 | ペルソナ戦略会議の進行・決議JSON生成(FR-3) | スケジュール起動(cron) |
| decision_gate | L1 | 決議JSONの検証(スキーマ/ホワイトリスト/変化幅)→ config 反映(FR-3.5) | 会議後に起動 |
| bot_runner | L1 | 戦略BOTの実行コンテナ。1BOT=1プロセス(FR-4) | 常駐 × BOT数 |
| risk_guard | L1 | 全BOT横断のリスク上限監視・サーキットブレーカ・キルスイッチ(FR-5) | 常駐(最優先) |
| executor | L1 | 取引所アダプタ経由の発注・約定照合・冪等性管理(FR-4.2) | bot_runner から呼出 |
| reporter | L1+L3 | KPI集計、週次/月次レポート生成のデータ準備(FR-6) | スケジュール起動 |
| feedback_runner | L3 | Claude Code ヘッドレス(`claude -p`)を起動し、レビューと改善提案を生成(FR-6.2〜6.3) | cron(週次/月次) |
| notifier | L1 | Discord Webhook 通知(FR-7.1) | ライブラリとして全体から利用 |
| watchdog | L1 | heartbeat 監視・自動再起動連携(FR-7.2) | systemd + 常駐スクリプト |

### 1.2 重要な設計判断とその理由

| 判断 | 理由 |
|---|---|
| 売買BOTは Claude Code 上で動かさず、独立した Python 常駐プロセスにする | Claude Code のヘッドレス実行は「1プロンプト実行して終了する」ワンショット型であり、24/7のデーモンではない。常駐が必要な売買・収集は systemd 管理の通常プロセスが適切。Claude Code は L3(開発・レビュー・改修)に専念させる |
| LLM呼び出しは Anthropic API を直接利用(L2) | 実行層から低レイテンシ・低コストで呼べる。tool use による JSON スキーマ強制で構造化出力を担保 |
| 決議は「config 差分」としてのみ反映 | LLM出力をコードに直結させない(要件 原則1)。差分はGit管理され、いつでもロールバック可能 |
| DBは SQLite(WAL) から開始 | 1台構成・書き込み頻度(秒〜分)なら十分。スキーマは SQLAlchemy で抽象化し、必要になったら PostgreSQL へ移行 |
| 通知は Discord Webhook | 無料・実装が数行・スマホ通知可。キルスイッチ操作もBotコマンドで拡張可能 |
| **抽象化は初日から、市場実装は縦切りで1つずつ** | 「あらゆる資産を扱える基盤」の最大の失敗様式は、抽象レイヤ作りに数ヶ月費やして実運用ループが1つも回らないこと。Instrument/Broker のインターフェース(§2.5)を Phase 0 で固定し、まず暗号資産で全層(収集→解析→会議→執行→レビュー)を貫通させてから、IBKR・国内株へ水平展開する |
| 2本目のアダプタは IBKR に集約 | 海外株・先物・債券ETF を1契約・1API(別途市場データ購読)でカバーでき、アダプタ実装数を最小化できるため。国内株は専用API(kabu STATION 等)が必要なため最後に回す |

### 1.3 主要フロー(ニュースイベント発生時の例)

```
RSS受信(news_collector)
  → 重複排除・正規化 → news_raw 保存
  → Stage1 ルールフィルタ(通過率を1〜2割に絞る)
  → Stage2 Haikuで分類 → news_signals 保存 {影響度, 銘柄, 方向, 確度, 持続時間}
  → 影響度≧4 なら Stage3 上位モデルで詳細分析
       ├→ リスクオフ提案 → decision_gate 検証 → risk_guard が新規停止/縮小
       └→ 臨時会議アジェンダ登録 → council_runner(臨時)
  → 各BOTは news_signals を自分のルールで参照
       例: news_drift BOT「影響度≧4・確度≧0.7・方向一致で小ロットエントリー、
           持続時間経過 or 逆行2%で決済」(全条件は決定的コード)
```

### 1.4 障害時の振る舞い(フェイルセーフの方向性)

| 障害 | 振る舞い |
|---|---|
| 市場データ鮮度切れ(FR-1.4) | 依存BOTは新規エントリー禁止。既存ポジションの決済ロジックのみ継続 |
| LLM API 障害/コスト上限到達 | 知能層を縮退。BOTは「最後に検証済みのconfig」で稼働継続(LLMがなくても売買は止まらない設計) |
| 取引所API障害 | 発注リトライ(指数バックオフ・上限あり)→ 失敗継続で当該BOT停止+通知 |
| プロセス異常終了 | systemd が再起動。起動時に取引所の建玉・注文と DB を突合し、不整合は人間へ通知して新規停止 |

## 1.5 ガバナンス設計(v0.3 追加 — 本フレームワークの中核)

> **v0.3 の再定義**: 本システムの一次成果物は特定の取引ルールではなく、**エージェント群がルールを提案・審議し、利用者が唯一の決裁権者としてトップダウンで決定し、決定だけがシステムに反映される**ガバナンス・フレームワークである。本書中の具体的な数値はすべて「たたき台」であり、正式値は 03_運営規程・第0回アジェンダ.md に基づく会議と決裁で定める。

### 1.5.1 権限モデル(三権分離)

| 権限 | 保有者 | 内容 |
|---|---|---|
| 提案権 | エージェント(会議・週次レビュー)、BOT実績データ(自動起案)、利用者 | **何でも提案できる**。リスク上限・レバレッジ・ゲート基準も対象。v0.2 の「会議はリスク設定に触れられない」制約は撤廃し、「提案は自由・適用は決裁のみ」に一本化 |
| 審議権 | 戦略会議(ペルソナ群) | 提案を多視点で揉み、決裁しやすい形(選択肢+推奨+根拠+リスク)に整える。risk_manager の veto は**審議段階の差し戻し**であり、決裁の代替ではない |
| 決裁権 | **利用者のみ** | 承認 / 修正承認 / 却下 / 会議差し戻し / 期限付き保留。委任(§1.5.4)も決裁によってのみ設定・変更・撤回できる |

### 1.5.2 不変条項(フレームワーク憲法 — 会議の議題にできない)

1. 決裁権は利用者のみに属する。決裁を経ずに本番挙動が変わる経路を実装しない
2. LLM非執行原則: エージェントは提案まで。執行・強制は決定的コードが行う
3. 全決定の監査ログ: 提案 → 審議 → 決裁 → 適用 → 結果 を欠落なく記録する
4. キルスイッチ: 利用者はいつでも全停止できる
5. **No Policy, No Trade(fail-closed)**: 必須ポリシーが未決裁・失効の領域では発注を拒否する

> 正直な注記: オーナーである利用者は、コードを直接書き換えれば技術的には何でもできる。不変条項の意味は「**エージェント・自動化・うっかりミスがこれらを迂回できない**」ことの構造的保証にある。利用者自身に対する自己拘束(大きな変更のクーリングオフ、二段階確認など)は任意機能として実装し、採否は P-01 の決裁で利用者自身が決める。

### 1.5.3 ポリシーレジストリ

すべての運用ルールを `config/policies/*.yaml` で管理し、システムは status=active のものだけを読む。

```yaml
# config/policies/P-03_account_risk.yaml(形式例)
policy_id: P-03
title: 口座リスク上限
status: active            # draft → proposed → approved → active → retired
version: 3
value:
  max_daily_loss_pct: 2.0
decision:
  decided_by: owner
  channel: sync_council    # sync_council | async_approve
  session_ref: "council-0 (第0回)"
  decided_at: 2026-06-22T21:00+09:00
effective_from: 2026-06-23
review_after: 2026-09-01   # 期限到来で自動的に会議へ再上程(FR-10.6)
```

- 既存の risk_limits.yaml / config/bots/*.yaml は、active なポリシーから**生成される実行用ビュー**と位置づける(手編集不可、生成スクリプト経由のみ。手編集は pre-commit で検出)
- DB に policies(現在値)/ policy_decisions(全決裁履歴)/ proposals(審議中)テーブルを追加
- ロールバック = 旧バージョンを再決裁して active に戻す(履歴は消さない)

### 1.5.4 決裁チャネル

| チャネル | 形式 | 主な用途 |
|---|---|---|
| **同期意思決定会議** | 利用者が議長のインタラクティブセッション。Claude Code の `/council`(§8.3)で開催し、ペルソナ達が審議、利用者がその場で決裁を宣言 → 決裁レコード+ポリシー差分を自動生成。リポジトリ整備前は claude.ai 上でも開催可能(03 §6) | 第0回(初期ポリシー策定)、月次見直し、大きな方針転換 |
| 非同期決裁 | 週次レビューの提案を Discord 要約で受領 → `make approve / reject / defer` | 日常の改善提案 |
| 委任(自動適用) | 利用者が P-01 で定義した範囲のみ decision_gate が自動適用し事後報告。**初期値: 委任なし(全件決裁)** | 軽微・高頻度の調整 |

### 1.5.5 decision_gate の再定義

v0.2 の「リスク設定に触れる提案は reject」を改め、**decision_gate = 委任ポリシー(P-01)の機械検証器**とする。入力された提案(会議決議・週次提案)を次の3つに振り分ける:
(a) 委任範囲内 → スキーマ・変化幅を検証して自動適用+事後報告 / (b) 範囲外 → **決裁キューへ回送**(破棄しない) / (c) 不変条項に抵触 → reject + 警告通知

## 2. 技術スタック(選定理由つき)

| 領域 | 採用 | 理由 |
|---|---|---|
| 言語 | Python 3.12 / asyncio | 取引・データ系ライブラリ資産が最大。Claude Code との相性も良い |
| 取引所接続 | ccxt + 取引所ネイティブWebSocket | 100以上の取引所を共通APIで抽象化(A-1/A-2 変更に強い)。低遅延が要る板情報のみネイティブWS |
| DB | SQLite(WAL モード)→ 将来 PostgreSQL | §1.2 参照。ORM は SQLAlchemy |
| LLM | Anthropic API。Stage2=Haiku系 / Stage3・日次=Sonnet系 / 週次会議=上位モデル | 役割ごとにモデルを使い分けてコスト制御(FR-7.3)。最新のモデル名・料金は公式ドキュメント(docs.claude.com)を実装時に確認 |
| 構造化出力 | tool use(JSONスキーマ強制)+ pydantic で受信側検証 | 「LLM出力は必ず検証を通す」原則の実装手段 |
| ニュース取得 | feedparser(RSS)、httpx | 無料ソース中心で開始。X API は有料のため Phase 2 で費用対効果を判断 |
| バックテスト | vectorbt または backtesting.py(+手数料・スリッページモデル必須) | 実装速度優先。ニュース戦略はイベントスタディ形式の自前検証も併用 |
| 通知 | Discord Webhook | §1.2 参照 |
| 実行基盤 | VPS(東京)1台 + systemd(+ 任意で Docker Compose) | 月額コスト最小構成。cron で L2/L3 のスケジュールジョブを起動 |
| 改善層 | Claude Code(対話 + ヘッドレス `claude -p`、サブエージェント、カスタムスラッシュコマンド、hooks) | §8 で詳細設計 |
| 秘密管理 | .env(direnv)+ pre-commit で漏えい検査 | 出金権限なしAPIキーのみ(FR-5.7) |

### 2.5 マルチアセット基盤設計(v0.2 追加 / FR-8, FR-9)

#### (1) 統一インストゥルメントモデル
全銘柄を `config/instruments/*.yaml` + DB の instruments テーブルで定義する。

```yaml
# config/instruments/n225_mini.yaml(例)
instrument_id: jpx.n225mini.fut
asset_class: futures          # crypto_spot | crypto_margin | equity_jp | equity_foreign | futures | fx | bond_etf
broker: ibkr                  # どのアダプタ経由で執行するか
symbol: "N225M"               # ブローカー側シンボル(アダプタが変換)
currency: JPY
tick_size: 5
lot_size: 1
session_calendar: jpx_futures # 立会・夜間セッション・休日の定義を参照
margin_rule: span_like        # 証拠金計算方式の識別子
```

BOT・リスク管理・会議・KPI はすべて `instrument_id` を介して動き、**資産クラス固有の知識はアダプタと margin_rule に閉じ込める**。

#### (2) ブローカー/データソース・マトリクス(日本居住の個人の現実解)

| 資産クラス | 第一候補アダプタ | データソース | レバレッジ上限(法令・2026/6時点) | 備考 |
|---|---|---|---|---|
| 暗号資産(現物) | 国内取引所(bitFlyer/GMO等, ccxt) | 取引所WS | 1倍(証拠金取引は個人2倍) | 24/7。**最初の縦切り対象** |
| 海外株 | IBKR(ib_async) | IBKR市場データ購読 | 信用相当 約2倍(Reg T) | 1契約で多資産をカバー |
| 先物(指数・金利等) | IBKR(国内先物は kabu STATION 等も候補) | 同上 | 証拠金制(実効数倍〜) | FR-5.10 ギャップ規則を必ず適用 |
| 債券系 | **債券ETF・金利先物で代替**(IBKR) | 同上 | ETFは現物/信用 | 個別債の個人向けAPI取引は非現実的(要件 §8-7) |
| 国内株 | kabu STATION API(auカブコム)等 | J-Quants API(JPX公式) | 信用 約3.3倍 | 国内株のAPIは選択肢が少ない。Phase 7 で採否判断 |
| FX(任意) | 国内FX業者API(OANDA等) | 業者API | 個人25倍(自主上限は10倍) | 10倍級レバの現実的な主戦場。採否は別途判断 |

> 注意: 各社のAPI提供状況・約款・規制は変わる。**各Phaseの着手時に必ず最新を確認**し、結果を docs/adr/ に記録すること。追加ライブラリ: `ib_async`(IBKR)、J-Quants クライアント(国内株データ)。

#### (3) セッション/カレンダー設計
- `core/market/calendar.py`: 取引所別の立会時間・夜間セッション・休日・SQ日を保持。スケジューラは instrument のセッション定義に従ってバー生成と発注可否を制御する
- 場外時間のシグナルは**キューせず破棄**し、翌セッション開始時に再判断する(古い判断での寄付き成行を防ぐ)
- 寄り直後 N 分(初期値: 5分)は新規エントリーを抑制(オープニングの乱高下回避)をデフォルトとする

#### (4) マルチ通貨・統合評価
- 口座残高は (broker, currency) 単位で管理し、リスク上限判定は **円建て統合評価額**(fx_rates で換算)に対して行う
- 外貨建て資産の為替変動損益は KPI 上で分離計上する(戦略の実力と為替を混ぜない)

#### (5) タイムホライズン対応(FR-9)
- bot_runner が2つの実行型を提供: **バー駆動型**(on_bar(1m〜1d) コールバック)と**スケジュール駆動型**(日次/週次/月次の rebalance() コールバック)
- 戦略会議の配分は `短期スリーブ / 中期スリーブ / 長期スリーブ` の3階層 → スリーブ内でBOT別配分(決議JSONスキーマ v2 で sleeve を追加)
- 長期スリーブは恒久レバレッジ1倍。KPI評価期間もスリーブ別(短期=週次、長期=月次〜四半期。FR-9.4)

## 4. データ設計(主要テーブル)

> 記法: **PK** = 主キー。すべてのテーブルに `created_at` を持つ。

| テーブル | 主要カラム | 用途 |
|---|---|---|
| candles | **(symbol, timeframe, ts)**, o, h, l, c, v | 市場データ(FR-1.1) |
| news_raw | **news_id**, source, url, title, body, published_at, fetched_at, dedup_hash | 原文保存。published_at と fetched_at の差 = 検知遅延(KPI) |
| news_signals | **signal_id**, news_id(FK), stage, impact(1-5), symbols[], direction, confidence, half_life_min, model, raw_json | Stage2/3 の出力(FR-2)。事後リターン列を後から付与し precision を算出 |
| council_sessions | **session_id**, kind(weekly/daily/adhoc), input_digest, started_at | 会議の開催記録(FR-3.6) |
| council_opinions | **opinion_id**, session_id(FK), persona, round, content_json | ペルソナ別発言。月次で的中率採点 |
| council_decisions | **decision_id**, session_id(FK), decision_json, gate_result(accepted/modified/rejected), applied_at | 決議と検証結果(FR-3.5) |
| bot_configs | **(bot_id, version)**, yaml_snapshot, changed_by(human/gate), decision_id(FK, nullable) | 設定変更の全履歴(監査性) |
| orders | **order_id**, bot_id, decision_id(FK), symbol, side, qty, price, type, status, exchange_order_id, idempotency_key | 全発注(FR-4.4)。根拠へ遡及可能 |
| fills | **fill_id**, order_id(FK), qty, price, fee, ts | 約定。スリッページ実測の元データ |
| positions | **(bot_id, symbol)**, qty, avg_price, updated_at | 現在建玉(再起動時に取引所と突合) |
| pnl_daily | **(bot_id, date)**, realized, unrealized, fees, equity | 日次損益。risk_guard の日次上限判定にも使用 |
| bot_kpi_weekly | **(bot_id, week)**, pf, sharpe, max_dd, win_rate, avg_r, trades, status | 週次KPI(FR-6)。悪BOT判定の入力 |
| incidents | **incident_id**, severity, component, summary, detail, resolved_at | 障害・異常の記録(ゲート条件「重大インシデント0」の根拠) |
| llm_usage | **call_id**, component, model, in_tokens, out_tokens, cost_est, ts | コストメーター(FR-7.3) |
| heartbeats | **(component, ts)** | 死活監視(FR-7.2) |
| instruments | **instrument_id**, asset_class, broker, symbol, currency, tick_size, lot_size, session_calendar, margin_rule | 統一インストゥルメントモデル(FR-8.1)。candles / orders / positions の銘柄参照はすべてこのIDに正規化する(v0.2) |
| accounts | **(broker, currency)**, balance, margin_used, updated_at | 通貨別口座残高。円建て統合評価は fx_rates と併用(FR-8.4) |
| fx_rates | **(pair, ts)**, rate | 統合評価・外貨換算用 |
| margin_snapshots | **(broker, ts)**, equity_jpy, margin_used_jpy, maintenance_ratio, effective_leverage | 証拠金・実効レバレッジ監視(FR-5.8〜5.9)の履歴 |

リレーションの背骨: `orders.decision_id → council_decisions / news_signals → news_raw` により、**「この注文はなぜ出たか」を必ず一次情報まで遡れる**。

## 5. エージェント設計(L2: 知能層)

### 5.1 ペルソナ定義(config/personas/*.yaml)

各ペルソナは「システムプロンプト+出力スキーマ+性格パラメータ」で定義する。例:

```yaml
# config/personas/risk_manager.yaml
id: risk_manager
model: weekly_tier          # モデル名は config/system.yaml で一元管理(モデル更新に強くする)
veto_power: true            # 拒否権(このペルソナのみ)
system_prompt: |
  あなたは保守的なリスクマネージャー。役割は収益機会の発見ではなく、
  破滅的損失の回避。以下を必ず審査する:
  - 提案された配分合計がエクスポージャー上限内か
  - 直近のドローダウン状況と整合するか
  - 今後7日の経済イベント(入力参照)に対する備えがあるか
  懸念があれば veto を行い、理由と代替案を必ず添える。
  リスク上限の変更も提案・審議の対象だが、適用には利用者の決裁が必須である。
  根拠データのない上限緩和案には veto(審議差し戻し)を行うこと。
output_schema: council_opinion_v1   # pydantic スキーマ名
```

初期5名の性格設計の要点:

| ペルソナ | 視点 | 役割上の偏り(意図的) |
|---|---|---|
| macro_analyst | 金利・規制・マクロイベント | 中期目線。週次の相場観の土台を作る |
| momentum_trader | トレンド・出来高 | 強気に偏りやすい設定。機会の取りこぼしを防ぐ |
| contrarian_value | 過熱感・乖離 | 弱気に偏りやすい設定。momentum への対抗軸 |
| quant_validator | データ・統計 | 「その主張はデータで裏付くか」を全員に問う。KPI・バックテスト結果を引用する義務 |
| risk_manager | 損失回避 | 拒否権。唯一「儲け」を評価基準に持たない |

> 意図: 全員が同じ方向を向くと多エージェントの意味がない。**偏りを設計してぶつける**ことで、単一モデル1回呼び出しより頑健な決議を狙う。ペルソナの的中率は council_opinions から月次で採点し、プロンプトを改修する(これも Claude Code のタスク)。

### 5.2 会議プロトコル(council_runner)

```
入力パッケージ作成(reporter が生成):
  市況サマリ(主要銘柄の週足/日足統計)+ 直近KPI + news_signals 上位 + 経済カレンダー7日分
Round 1: 各ペルソナが並列で意見JSONを提出(他者の意見は見えない)
Round 2: 全員の意見を共有し、各自1回だけ反論・修正
集約   : 議長(facilitator)が決議JSON草案を作成
審査   : risk_manager が承認 / 修正要求 / veto
出力   : council_decisions へ保存 → decision_gate へ
```

- 週次会議: 全ラウンド実施(上位モデル)。日次ブリーフ: Round 1 と集約のみ(Sonnet系)。臨時: Stage 3 イベント時に該当銘柄スコープで縮小開催
- 呼び出し回数の目安: 週次 = 5名×2 + 議長 + 審査 ≒ 13回/週、日次 ≒ 7回/日 → コストメーターで実測し A-6 内に収める

### 5.3 決議JSONスキーマ(v1・抜粋)

```json
{
  "session_id": "2026-W24",
  "market_outlook": { "bias": "neutral", "confidence": 0.55, "summary": "..." },
  "allocations": [
    { "bot_id": "trend_btc", "weight": 0.40 },
    { "bot_id": "news_drift", "weight": 0.15 }
  ],
  "param_changes": [
    { "bot_id": "range_eth", "key": "entry_band_sigma", "from": 2.0, "to": 2.5, "rationale": "..." }
  ],
  "new_strategy_candidates": [ { "name": "...", "hypothesis": "...", "data_needed": "..." } ],
  "risk_flags": [
    { "level": "warn", "window": "2026-06-17/18", "reason": "FOMC", "action": "reduce_exposure_50" }
  ],
  "veto": null
}
```

### 5.4 decision_gate の検証規則(L1・決定的)

| 規則 | 内容 |
|---|---|
| スキーマ検証 | pydantic で型・必須項目・値域を検証。失敗は全体 reject |
| ホワイトリスト | param_changes はBOTごとに「変更可能キー一覧」に載っているものだけ受理 |
| 変化幅制限 | 配分 weight の週次変化 ≤ ±15pt、各パラメータの変化 ≤ 定義済み上限。超過分はクリップして適用し、その旨を記録 |
| 委任判定 | 委任ポリシー(P-01)の範囲内か判定。範囲内のみ自動適用、範囲外(リスク上限・レバレッジ等を含む)は**決裁キューへ回送**(§1.5.5) |
| 不可侵領域 | 不変条項(§1.5.2)に抵触する提案は無条件 reject + 警告通知 |
| 適用 | 受理分のみ config/bots/*.yaml に diff 適用 → bot_configs に履歴保存 → Git commit |

### 5.5 ニュース解析プロンプトI/F(Stage 2 の例)

- 入力: タイトル+本文先頭1,000字+ソース名+対象銘柄辞書
- 出力(tool use で強制): `{"impact": 1-5, "symbols": [...], "direction": "bullish|bearish|unclear", "confidence": 0-1, "half_life_min": int, "one_line_reason": str}`
- 較正: 週次レビューで「impact別の事後リターン分布」を集計し、プロンプト・閾値を更新(= ニュースBOT自体もフィードバックループの対象)

## 6. リスク管理設計(L1: risk_guard)

### 6.1 risk_limits.yaml(初期値の例)

```yaml
account:
  max_daily_loss_pct: 2.0        # 到達で当日全BOT新規停止(FR-5.2)
  max_weekly_drawdown_pct: 5.0   # 到達で全停止+人間呼び出し(FR-5.3)
  max_total_exposure_pct: 60.0   # 口座残高に対する建玉合計上限
per_trade:
  max_loss_pct: 1.0              # 1取引の許容損失(ポジションサイズ算出の根拠)
per_bot:
  max_weight: 0.5                # 単一BOTへの配分上限
  max_positions: 3
circuit_breaker:
  price_jump_pct_1m: 5.0         # 1分で5%超変動 → 新規停止15分
  max_spread_bps: 30
  stale_data_sec: 90
kill_switch:
  flag_file: /var/run/tradecouncil/KILL
  close_positions_on_kill: ask   # ask | always | never
```

### 6.2 実装上の要点

1. risk_guard は executor の**手前**に位置し、全注文がここを通る(BOTが直接取引所を呼ぶ経路を作らない)
2. 上限判定は「約定ベースの実現損益+評価損益」で行い、pnl_daily と突合
3. キルスイッチはフラグファイル方式(`touch` 1つで停止できる)+ Discord コマンド + `make kill`
4. risk_limits.yaml を含む全ポリシーの変更は**決裁レコード必須**(§1.5.3)。提案は会議・週次レビューから自由に出せるが、適用は決裁チャネル経由のみ。risk/ 関連は risk-auditor による機械チェックを併用する
5. tests/risk/ に「上限を超える注文が必ず弾かれる」性質テストを置く(回帰の絶対防衛線)

### 6.3 レバレッジ・証拠金管理(v0.2 追加 / FR-5.8〜5.10)

risk_limits.yaml に leverage / margin / overnight セクションを追加する:

```yaml
leverage:
  account_max_effective: 1.0     # 口座全体の実効レバレッジ上限(ゲート通過で段階引き上げ)
  per_asset_class:               # クラス別上限。0 = 使用禁止
    crypto_spot: 1.0
    crypto_margin: 0.0
    equity_foreign: 1.0
    equity_jp: 0.0               # アダプタ未実装クラスは0で封鎖
    futures: 0.0
    fx: 0.0
    bond_etf: 1.0
  hard_ceiling: 10.0             # 自主上限。法令上限がこれより低いクラスは法令上限が優先
margin:
  warn_maintenance_ratio: 3.0    # 維持率300%で警告通知
  reduce_maintenance_ratio: 2.0  # 200%で新規停止+段階的縮小開始
overnight:
  session_markets_max_factor: 0.5  # 立会時間制市場は引け前に実効レバを上限の50%へ自動縮小
```

運用規則:
1. **実効レバレッジ** = Σ|建玉想定元本(円建て)| ÷ 円建て統合評価額。risk_guard が常時計算し margin_snapshots に記録
2. 解禁は**ゲート制**: 各クラス「1倍で8週稼働・PF>1.2・最大DD<10%・証拠金インシデント0」→ 次段階(例: 2倍)へ。以後も同条件で1段階ずつ。上限は常に「法令上限と hard_ceiling(10倍) の小さい方」
3. 長期スリーブの建玉は実効レバレッジ計算上も恒久1倍枠で管理(FR-9.3)
4. ロスカットをブローカー任せにしない: reduce しきい値は各社の強制ロスカット水準より十分手前に設定する
5. 本節の数値はすべて**たたき台(P-02〜P-04)**であり、第0回会議の決裁で確定する。決裁されるまでフレームワークは fail-closed として動作し、未決裁クラスでは発注しない(不変条項5)

## 7. フィードバックループ設計(L3: 週次/月次)

### 7.1 週次サイクル(毎週月曜 07:00 JST、cron)

```
1. feedback/kpi.py が bot_kpi_weekly・news_signals 採点・ペルソナ採点を集計
2. claude -p(ヘッドレスモード)を起動し、feedback/prompts/weekly_review.md を実行
   入力: KPI集計 + 今週の incidents + council_decisions と実績の突合
   出力: reports/weekly/2026-W24.md(レビューレポート)
         feedback/proposals/2026-W24/*.diff(config変更提案・必要ならコード改修のPR下書き)
   権限: 読み取り+reports/proposals 配下への書き込みのみに制限(本番configへの直接書込は不可)
3. notifier がレポート要約と提案一覧を Discord へ
4. **決裁権者(利用者)**が提案を確認 → `make approve PROPOSAL=...`(=決裁行為。決裁レコードを自動生成し Git commit)で適用 / 却下 / 会議付託
5. 悪BOT判定(FR-6.4)は kpi.py が機械的に実施し、状態遷移を自動適用:
   ACTIVE → REDUCED → PAPER → RETIRED(各遷移を通知。復帰は月次で人間が判断)
```

### 7.2 月次サイクル(月初)

- 戦略ポートフォリオの見直し: RETIRED 候補の確定、PAPER からの復帰審査、新戦略候補(決議JSON由来)の優先順位付け
- 採択された新戦略は「課題票(hypothesis / 検証データ / 合格基準)」として docs/adr/ に発行し、Claude Code の実装タスクにする
- ペルソナ採点に基づくプロンプト改修、ニュースStage2の閾値較正

### 7.3 自動化の段階

| 段階 | 内容 |
|---|---|
| 初期(Phase 4) | 提案生成まで自動。適用は100%人間承認 |
| 中期 | 「変化幅が小さい・実績が安定」の提案カテゴリのみ自動適用を検討(decision_gate と同じ検証を通す) |
| 当面やらない | コード変更の無承認デプロイ。リスク上限の自動変更(恒久的に禁止) |

## 8. Claude Code への引き継ぎ設計(L3)

### 8.1 CLAUDE.md

リポジトリ直下に置くプロジェクトメモリ。**別ファイル CLAUDE.md としてドラフトを同梱**した。要点: プロジェクト概要 / 絶対ルール(安全規約)/ アーキテクチャ要約 / よく使うコマンド / 変更禁止領域。

### 8.2 サブエージェント(.claude/agents/)

| サブエージェント | 役割 | ツール権限の方針 |
|---|---|---|
| risk-auditor | risk/ や risk_limits.yaml に触れる差分の安全審査(上限緩和・経路バイパスの検出) | 読み取り専用 |
| backtest-runner | 戦略のバックテスト実行と結果要約(`make backtest` の実行のみ許可) | Bash(限定)+読み取り |
| weekly-analyst | 週次レビューの分析本体(feedback_runner から委譲) | 読み取り+reports/proposals への書き込み |
| code-reviewer | 一般的なコードレビュー(冪等性・例外処理・秘密情報の混入) | 読み取り専用 |

> サブエージェントは description を見て自動委譲されるため、各定義の description に「いつ使うか」を明確に書く。軽い調査系は安価なモデルに割り当ててコストを抑える。

### 8.3 カスタムスラッシュコマンド(.claude/commands/)

| コマンド | 内容 |
|---|---|
| /weekly-review | §7.1 の手順を対話セッションで手動実行(cron 障害時の代替) |
| /backtest <bot_id> | 指定戦略のバックテスト→合格基準(§9)との照合レポート |
| /risk-audit | 現在の差分に対して risk-auditor を起動 |
| /new-strategy <name> | 課題票テンプレートから戦略の雛形(bots/ + config + テスト)を生成 |
| /council <議題> | **同期意思決定会議の開催**: 利用者が議長・決裁権者。会議パッケージ(市況/KPI/提案一覧)を提示し、ペルソナ達が審議、利用者の決裁宣言で決裁レコード+ポリシー差分(config/policies/)を生成する(§1.5.4)。第0回は `/council 第0回` で 03 のアジェンダを読み込んで開催 |

### 8.4 hooks(.claude/settings.json)

- PostToolUse(Edit/Write): `make test-fast` を自動実行し、編集が既存テストを壊していないか即時確認
- PreToolUse(Bash): `make live` など実弾系コマンドをブロック(実弾操作は人間の手でのみ実行)
- pre-commit(Git側): 秘密情報スキャンと risk_limits.yaml 変更の警告

### 8.5 引き継ぎ手順(このドキュメント一式を渡した後)

```
1. 空リポジトリに docs/ と CLAUDE.md を配置して git init
2. Claude Code を起動し、最初の指示:
   「docs/01_要件定義書.md と docs/02_基本設計書.md を読み、
    §9 Phase 0 のスコープでリポジトリ骨格と risk_guard・paper executor を
    実装してください。まず計画(plan)を提示し、承認後に着手すること。
    risk/ 配下はテストを先に書くこと。」
3. Phase 0 完了条件(§9)を満たしたら、ペーパーBOT 1体の24h稼働試験へ
```

> 補足: Claude Code の機能(サブエージェント・hooks・ヘッドレス等)の最新仕様は実装時に公式ドキュメント https://code.claude.com/docs を参照すること(本書の記述は2026年6月時点)。

## 9. 開発フェーズ計画とゲート条件

| Phase | スコープ | 完了条件(DoD) |
|---|---|---|

| 1. 単一戦略(2〜4週) | market_collector、トレンドフォロー1本、バックテスト環境、日次サマリ通知 | バックテスト合格(PF>1.2 / 最大DD<15% / 取引≥100、手数料込み)→ ペーパー継続稼働 |
| 2. ニュース(2〜3週) | news_collector、3段パイプライン、news_drift BOT(ペーパー)、リスクオフ防御BOT | 検知レイテンシ p50<30秒。impact別の事後リターン集計が出る(精度はこの時点で問わない) |
