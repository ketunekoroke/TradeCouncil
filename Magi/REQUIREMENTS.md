# 要件一覧 — Magi(汎用マルチエージェント・シナリオ基盤)

Magi プロジェクト固有の要件。**一次資料は [docs/07_シナリオ・人格基盤.md](docs/07_シナリオ・人格基盤.md)**。
共通ブリッジ(LLMバックエンド選択・ファイル入力・SharePoint・非機能)の要件は
[../shared/REQUIREMENTS.md](../shared/REQUIREMENTS.md)、モノレポ全体は
[../REQUIREMENTS.md](../REQUIREMENTS.md) を参照。実装は [FEATURES.md](FEATURES.md)、検証は
[TESTCASES.md](TESTCASES.md)。

- 優先度: **必須** / **推奨** / **任意** ／ 状態: **実装済** / **仕様**(プロトコルとして定義・AI が手動実行)

## SC. シナリオ・アーキテクチャ

| ID | 要件 | 優先度 | 状態 |
|---|---|---|---|
| REQ-SC01 | ルーター(CLAUDE.md)がモード判定・シナリオ選択を行う | 必須 | 仕様 |
| REQ-SC02 | 各シナリオは独立した1ファイル(scenarios/<name>.md)。共通作法は CLAUDE.md を唯一の出典とする | 必須 | 仕様 |
| REQ-SC03 | 人格ごとに LLM バックエンド(claude/openai/gemini)を frontmatter で切替できる | 必須 | 実装済(ブリッジ) |
| REQ-SC04 | 4シナリオ(合議/資料レビュー/ブレスト/人格テスト)がプロジェクト dir で実行できる | 推奨 | 仕様 |
| REQ-SC05 | シナリオごとに出力先を分離(workspace/<種別>)。root は単一の workspace/ で、SharePoint 連携時は全フォルダが双方向自動同期される(追加型・newer-wins・削除非伝播 — ADR-0009) | 推奨 | 実装済 |
| REQ-SC07 | 成果物: 議論ログ等を Markdown で常に生成する。Excel/Word/チャートは任意生成 | 必須/任意 | 仕様 |
| REQ-SC08 | どの人格がどの backend/model で動いたかを各成果物の冒頭に明記する(再現性。フォールバック発火時は実際のモデル) | 推奨 | 仕様 |

|---|---|---|---|
| REQ-DL01 | 価値観で分かれた3人格(MELCHIOR=論理 / BALTHASAR=関係性 / CASPER=直感)が議論する | 必須 | 仕様 |
| REQ-DL02 | ファシリテーター(メインセッション)が論題整理・進行・合意形成・成果物生成を担う | 必須 | 仕様 |
| REQ-DL03 | 議題の重要度に応じ Lite / Standard / Full の3モードを選べる | 必須 | 仕様 |

## DR. 資料チェック&リバイス シナリオ(document-review — 一次: docs/07・scenarios/document-review.md)

| ID | 要件 | 優先度 | 状態 |
|---|---|---|---|
| REQ-DR01 | 3人格の価値観レンズをレビュー観点に投影する(MELCHIOR=正確性/論理 / BALTHASAR=読者/伝わりやすさ / CASPER=訴求力/声)。専門の編集者・校正者には作り変えない | 必須 | 仕様 |
| REQ-DR02 | Round 0 で資料の素性(種類/目的/対象読者/トーン)と改訂の制約(不可変箇所・分量・リライト強度)を確定する | 必須 | 仕様 |
| REQ-DR03 | 各人格が同じ資料を独立にレビューし、重要度・箇所・問題・修正提案つきで指摘する | 必須 | 仕様 |
| REQ-DR04 | 深度モード Quick / Standard / Deep を選べる(相互レビュー・優先度合意・再検証の有無が変わる) | 必須 | 仕様 |
| REQ-DR05 | 衝突する指摘をファシリテーターが裁定し、must/should/nice/見送りに分類する | 必須 | 仕様 |
| REQ-DR06 | 合意した指摘を反映した改訂版を**元と同形式**で生成し、変更履歴を記録する | 必須 | 仕様 |
| REQ-DR07 | 成果物として指摘レポートと改訂版の両方を出す(指摘だけ・改訂だけにしない) | 必須 | 仕様 |
| REQ-DR08 | 未採用の指摘とその理由を成果物に残す(少数意見の保持に相当) | 推奨 | 仕様 |
| REQ-DR09 | 原文を尊重し、指摘を捏造せず、合意範囲を超えた全面リライトをしない | 必須 | 仕様 |
| REQ-DR10 | docx 等書式付き資料の改訂版は原則「原本コピー編集」で体裁・画像を保ち、全面再構築は構造変更時の最終手段とする(失う体裁を Round 0 で確認) | 推奨 | 仕様 |

## BR. ブレストシナリオ(brainstorm — 一次: docs/07・scenarios/brainstorm.md)

| ID | 要件 | 優先度 | 状態 |
|---|---|---|---|
| REQ-BR01 | 3人格の価値観レンズを**アイデアの発散と評価の両方**に投影する(MELCHIOR=実現可能/体系 / BALTHASAR=人中心/持続 / CASPER=大胆/独創)。専門の企画者・コンサルには作り変えない | 必須 | 仕様 |
| REQ-BR02 | Round 0 で目的・制約・成功基準・評価軸(3〜6)と発散ルール(判断後回し/量優先/突飛歓迎/build-on)を確定する | 必須 | 仕様 |
| REQ-BR03 | モード Quick / Standard / Deep を選べる(発散巡数・プレモーテムの有無が変わる)。発散巡数はモード固定で、新案が尽きたら早期収束する | 必須 | 仕様 |
| REQ-BR04 | 各人格が独立に大量発散し(質より量)、アイデアマップ化(クラスタ化・白地の明示)してから build-on・掛け合わせ・空白埋めで二次発散する | 必須 | 仕様 |
| REQ-BR05 | ショートリストの各アイデアを全人格が評価軸で 0〜10 採点し一言根拠を付す | 必須 | 仕様 |
| REQ-BR06 | スコア集計はレンズ別内訳を保持し(単純平均で潰さない)、上位 N に加えて評価が割れた尖り案を別枠で残す | 必須 | 仕様 |
| REQ-BR07 | 上位案を3レンズ協働でブラッシュアップ(CASPER訴求×MELCHIOR実現×BALTHASAR人/持続)。Deep ではプレモーテムで事前ストレステスト | 必須 | 仕様 |
| REQ-BR08 | 成果物としてアイデア集・アイデアマップ(Mermaid)・評価マトリクス・上位案を出す。アイデア・スコアを捏造しない | 必須 | 仕様 |
| REQ-BR09 | 発散と収束を分離する(発散フェーズでは批判させない)。意図的な弱みを保持し、レンズの差を発散にも評価にも残す | 推奨 | 仕様 |

## PT. 人格テストシナリオ(persona-test — 一次: docs/07・scenarios/persona-test.md)

| ID | 要件 | 優先度 | 状態 |
|---|---|---|---|
| REQ-PT01 | 同一のプローブ(依頼)を全人格に等しく独立に投げ、互いの出力を見せずに答えさせる(議論させない=個性を分離) | 必須 | 仕様 |
| REQ-PT02 | 固定のプローブ・バッテリ(判断 / 好奇心 / 弱み / ドメイン非依存 / 反応)を持ち、回帰テストとして再現可能にする(ユーザーは追加可、既定は不変) | 必須 | 仕様 |
| REQ-PT03 | 各人格の定義(レンズ / 好奇心の向き / 意図的な弱み / 声)をルーブリックに、期待と照合して人格別に判定する。弱みが出るのは仕様どおりと評価する | 必須 | 仕様 |
| REQ-PT04 | 識別性チェック(2人格が似すぎ=個性の潰れ・混ざりを検出して警告する) | 必須 | 仕様 |
| REQ-PT05 | backend は既定で各人格の設定どおり(混在可)、任意で同一 backend に揃えられる(混在時は出力差にモデル差が混じりうると注記) | 推奨 | 仕様 |
| REQ-PT06 | モード Quick / Standard / Deep を選べる(プローブ範囲・複数 backend・ベースライン回帰の有無が変わる) | 推奨 | 仕様 |
| REQ-PT07 | (任意)過去の実行をベースラインに差分比較し、調整で狙った変化が出たか・他の特性が壊れていないかを確認する | 任意 | 仕様 |
| REQ-PT08 | 成果物として使用した設問・差分マトリクス・人格別判定を出す。出力を期待に寄せて歪めず、中身・判定を捏造しない | 必須 | 仕様 |

## PE. 人格定義(一次: docs/07)

| ID | 要件 | 優先度 | 状態 |
|---|---|---|---|
| REQ-PE01 | 人格は専門分野ではなく価値観・判断軸で定義する(専門ペルソナにしない) | 必須 | 実装済 |
| REQ-PE02 | 人格は frontmatter に `name` / `description` / `backend` / `model` を持つ | 必須 | 実装済 |
| REQ-PE03 | 人格定義の本文がそのままシステムプロンプトになる(フロントマター・メタコメントは除外) | 必須 | 実装済 |
| REQ-PE04 | 3人格は共通の「好奇心・興味」を持ち、向く対象と強度がレンズごとに屈折する(向きにくい対象は「意図的な弱み」と呼応。弱みは削らない) | 推奨 | 実装済 |
