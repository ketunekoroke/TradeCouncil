# 戦略開発ガイド

新しい BOT 戦略を作るときの標準フロー。テストファースト・ドキュメント駆動
(docs/05 §1)を戦略開発に具体化したもの。ノウハウの蓄積先は
[戦略カタログ(docs/strategies/)](strategies/README.md)、判断の経緯は
[ADR-0007](adr/0007-strategy-knowledge-management.md)。

---

## 1. 標準フロー(6ステップ)

```
① カード先行 → ② tc bot new → ③ レジストリ登録 → ④ テスト先行で実装 → ⑤ 試走 → ⑥ 学びを追記
```

### ① 戦略カードを先に書く(ドキュメント駆動)

実装より先に**仮説と合格基準**を書く。会議で採択された新戦略の課題票も
`status: draft` のカードとして発行する(ADR-0007 §2)。
最低限「仮説」「合格基準」が書けないなら、まだ実装するべきではない。

### ② `tc bot new` で雛形を一括生成する

```
.venv\Scripts\python.exe -m scripts.cli bot new <bot_id> --strategy <strategy_key>
# 例:
.venv\Scripts\python.exe -m scripts.cli bot new ma_cross_btc --strategy ma_cross
```

生成されるもの(既存ファイルがあれば**何も書かずに拒否**):

| ファイル | 内容 |
|---|---|
| `bots/<strategy_key>.py` | Strategy サブクラスの雛形(rationale-base パターン入り) |
| `config/bots/<bot_id>.yaml` | BOT 設定。**`enabled: false` 既定**(意図せず走らせない) |
| `tests/bots/test_<strategy_key>.py` | on_bar 単体テスト雛形 + レジストリ登録検査 |
| `docs/strategies/<strategy_key>.md` | 戦略カード(`_template.md` から生成) |

①でカードを書いた後に②を実行した場合は、生成されたカードに①の内容を移す
(②を先に実行して生成カードへ直接書いてもよい)。

### ③ `STRATEGIES` レジストリに登録する

スキャフォールドは `bots/__init__.py` を**自動編集しない**。生成されたテストの
レジストリ登録検査が red になるので、登録して green にする:

```python
# bots/__init__.py
from bots.ma_cross import MaCross

STRATEGIES: dict[str, type[Strategy]] = {
    "dummy_random_walk": DummyRandomWalk,
    "ma_cross": MaCross,          # ← 追加
}
```

> 自動編集しないのは、実行可能な戦略の追加を人間のレビューを伴うコード変更として
> 扱うため(ADR-0007 §4)。

### ④ テスト先行でロジックを実装する

生成されたテスト雛形に on_bar の期待挙動を先に書き(red)、戦略を実装して
green にする。`python -m scripts.cli test` が全緑になるまで完了と言わない。

### ⑤ サンドボックスで試走する

```powershell
$env:TC_VAR_DIR = "var-sandbox"      # 本番 var/ を汚さない(docs/05 §3.3)
.venv\Scripts\python.exe -m scripts.cli db init
.venv\Scripts\python.exe -m scripts.cli paper --bot <bot_id>   # 要 enabled: true
```

config の `enabled: false` を試走時に true へ変更し、終わったらサンドボックス
ディレクトリごと削除する。本番の config/bots/ で enabled を切り替える変更は
通常のコードレビュー(コミット)を経る。

### ⑥ 学びをカードに追記する(append-only)

試走・検証で得た学び(効いた/効かなかったパラメータ、想定外の挙動)を
カードの「運用での学び」に**日付付きで追記**する。検証結果の解釈は
「検証結果」表へ。実績数値は転記しない(DB が真実源 — `tc kpi`)。

ゲート進行(backtest → paper → live)は P-06 ポリシーの決裁に従う。
Phase 0〜4 は paper まで(live 系は存在しない)。

## 2. 戦略実装の規約

[bots/base.py](../bots/base.py) の `Strategy` を継承し `on_bar` を実装する。

```python
class MyStrategy(Strategy):
    def __init__(self, bot_id: str, params: dict) -> None:
        super().__init__(bot_id, params)
        self._lookback = int(params["lookback"])   # パラメータは YAML から(ハードコード禁止)

    def on_bar(self, bar: BarData, position_qty: float) -> list[StrategyIntent]:
        ...
```

**守ること**:

1. **bots/ から `core.exchange` / `core.execution` を import しない**(原則2:
   権限分離)。戦略は BarData → StrategyIntent の純粋な変換のみ。発注経路
   (根拠起票 → risk_guard → executor)は bot_runner が握る。
   この制約は `tests/risk/test_limits.py::TestNoBotExchangeImport` で機械検査される
2. **rationale を必ず埋める**: 全 StrategyIntent の `rationale` に判断材料
   (rule 名・bar_ts・close・position_qty + 戦略固有のシグナル値)を入れる。
   これが trade_decisions に記録され週次レビューの一次資料になる(FR-6)
3. **est_max_loss_jpy をロジックから導出する**: ストップ幅・想定スリッページ等から
   見積もる。risk_guard の損失上限チェックの入力になる(過小見積もりは
   リスク管理の穴になる)
4. **パラメータは `config/bots/<bot_id>.yaml` の `params`** に置く。戦略コードに
   数値をハードコードしない。値の根拠はカードの「パラメータと根拠」表に書く
5. **決定的に書く**: 同じバー列に対して同じ意図を返す(乱数を使うなら seed を
   params で受ける)。テスト・検証の再現性のため
6. **状態は自分で持ってよいがプロセス再起動で消える前提**で設計する
   (永続化が必要な状態はまだサポートしない — 必要になったら ADR を起票)

## 3. よくある落とし穴

| 落とし穴 | 正しいやり方 |
|---|---|
| 雛形生成後にレジストリ登録を忘れて `paper` 起動が「未登録の戦略」で落ちる | 生成されたテスト(登録検査)を先に流す — red なら登録漏れ |
| est_max_loss_jpy に適当な固定値を入れる | ロジックから導出。ダミー戦略の固定 1% は「試験用だから許される手抜き」(カード参照) |
| カードを書かずに実装だけ進める | カード先行(①)。仮説が書けない戦略は検証もできない |
| 学びを Slack/会話だけに残す | カードに append-only で追記。会議でペルソナが引用できる形がノウハウ |
| 退役した戦略のカードを消す | `status: retired` にして退役理由を追記(同じ失敗を繰り返さない) |
| 試走のために本番 var/ で DB を作り直す | `TC_VAR_DIR` サンドボックス(⑤)を使う |
