## 背景

通常のシステム開発では「テスト環境と本番環境の常設分離」が定石だが、本プロジェクトは
AI(Claude Code)が開発主体であり、**AI がテスト・本番両方の実態(DB・ログ・稼働状況)を
確認できる**ことが品質の前提になる。また個人運用(A-6: 月1万円上限)であり、
常設環境を増やすコストは LLM 予算を直接圧迫する。
「本番のみで開発するか、テスト環境を毎回作って壊すか」を検討した。

## 決定事項

### 1. テスト環境は常設せず、3層で代替する

①単体/E2E = pytest の使い捨て環境(既存)②結合確認 = `TC_VAR_DIR` サンドボックス
(作って壊す。SQLite+ファイル構成のため環境の実体はディレクトリ1つでコスト≒0)
③恒久ステージング = **paper 運用そのもの**(本番と同一コード・同一ポリシー・同一経路で
常時走り、Phase 5 以降も live と並走する)。
**「本番のみで開発」は却下**: 作業ツリーと運用環境の分離は git が担い、
本番に未コミットの変更を置かない。

### 2. TC_VAR_DIR による実行時生成物の差し替え(サンドボックス)

- 環境変数 `TC_VAR_DIR` で `var/` 配下(DB・キルフラグ・ログ)一式を別ディレクトリへ
  読み替える。system.yaml は無変更、未設定時は完全に従来挙動
- 実装: `core/config.py` の `resolve_runtime_path()` が yaml 値の `var/` 接頭辞のみを
  読み替える(サブ構造 `run/`・`logs/` は維持)。環境変数は**プロパティ評価時に動的に**読む
- **config/(ポリシー)と `.env` は共有** — fail-closed・決裁状態が本番と一致したまま
  検証できることを意図する
- 制約: DB エンジンはプロセス内キャッシュのため、`TC_VAR_DIR` は**プロセス起動前に確定**
- 命名規約 `var-<name>`(gitignore `/var-*/`・hooks の手編集保護対象)
- 却下案: 環境ごとの system.yaml 複製(設定ドリフトの温床)/ config プロファイル機構
  (Phase 0 には過剰。必要になれば再検討)

### 3. AI のアクセスモデル — 観測と変更の非対称

- **観測(read)= 本番直接可**: DB・ログ・`tc status`/`kpi`・Teams 通知
- **変更(write)= 必ず git + 決裁ゲート経由**: 本番での直接編集は禁止
  (コード = 手元開発 → git → デプロイ / 運用ルール = `tc policy record` のみ)
- これは LLM 非執行原則(不変条項2)の環境への拡張である

### 4. 実弾(Phase 5)は同一サーバに live + paper 同居

- 防御はソフトウェア層で担う: 出金不可 API キー / risk_guard の型レベル経路遮断 /
  fail-closed / hooks(block_live)/ キルスイッチ / decision_id 遡及監査
- 再検討トリガー: 実弾資金が会議で定める閾値を超える、または live 系インシデント発生時に
  live 専用サーバ分離を会議へ付議する
- 却下案: 最初から live 専用サーバ分離 — +1台のコストと観測・デプロイの2系統化に対し、
  Phase 5 開始時点(少額1BOT)ではソフトウェア防御で釣り合うと判断

### 5. Claude Code は「手元開発 + サーバ同居(用途限定)」

- 開発は手元 Windows の Claude Code。サーバ上は ①`claude -p` 定型ジョブ(Phase 4)
  ②障害調査の SSH 対話(観測のみ)に限定し、本番ツリーの直接編集・コミットは禁止
- hooks はサーバでも同一設定で適用する

### 6. ホスティング(A-5)は Phase 1 着手時に決定

- 選択肢(国内 VPS / Azure VM / 自宅)と判断基準は docs/05 §4 の比較表に保持。
  決定時に docs/01 A-5 を確定し、ADR を起票する(BACKLOG: BL-016)
