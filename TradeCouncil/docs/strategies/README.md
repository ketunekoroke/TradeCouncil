# 戦略カタログ(Strategy Catalog)

BOT 戦略の**ノウハウを蓄積・再利用するための知識ベース**。1戦略 = 1カード
(`<strategy_key>.md`)。新規カードは `_template.md` から作る(`tc bot new` が自動生成)。
位置づけ・運用原則は [ADR-0007](../adr/0007-strategy-knowledge-management.md)、
開発フローは [docs/06_戦略開発ガイド.md](../06_戦略開発ガイド.md) を参照。

> 新しいカードを作ったら必ずこの表に1行追加する(索引が真実源)。

## 状態(status)とゲート

カードの `status` は P-06(フェーズゲート基準)の段階と対応する:

```
draft → backtest → paper → live → retired
```

| status | 意味 | 進む条件 |
|---|---|---|
| `draft` | 課題票段階(仮説・合格基準のみ。コード未完成も可) | — |
| `backtest` | 実装済み・バックテスト検証中 | テスト緑 + カードの仮説・合格基準が記入済み |
| `paper` | ペーパー稼働中 | P-06 のバックテスト基準を満たす(ゲート進行は決裁事項) |
| `live` | 実弾稼働中(Phase 5 以降。Phase 0 には存在しない) | P-06 のペーパー基準 + 決裁 |
| `retired` | 退役(悪BOT判定 or 手動)。**カードは消さない** | 月次レビュー(FR-6.3〜6.5) |

## 運用ルール

1. **カード先行(ドキュメント駆動)**: 実装より先に仮説・合格基準を書く。
   会議で採択された新戦略の課題票も `status: draft` のカードとして発行する
2. **「運用での学び」は append-only**: 日付付きで追記し、消さない・上書きしない。
   失敗の記録(効かなかったパラメータ・想定外の挙動・退役理由)こそ次の戦略の資産
3. **数値は DB、解釈はカード**: 実績数値(損益・勝率・DD)は DB が真実源
   (`tc kpi` / bot_kpi_weekly)。カードには数値を転記せず、評価の解釈・学び・
   次のアクションを書く(転記すると必ず陳腐化する)
4. **会議の知識源**: council / deliberation でペルソナが引用できる。召喚時に
   関連カードのパスを渡してよい(全員に等しく同じものを渡す)
5. **秘密情報を書かない**: API キー・残高・Webhook URL 等は書かない(git 追跡される)
6. **retired でも残す**: 退役理由と学びを追記して残す。同じ失敗を繰り返さないための記録
