# 議題 P-01「決裁・委任規程」★

- たたき台: 委任なし(全件決裁)。自己拘束機能(クーリングオフ等)は不採用
- 必須キー: `delegation.enabled`
- 論点(docs/03): どこまで自動適用を許すか / 大変更に自己拘束を課すか

## Round 1: ペルソナ意見(並列・独立)

### MACRO_ANALYST(claude/opus)

The implementation confirms the key facts: P-01 changes themselves are never auto-applied (always decided by owner), delegation scopes are per-policy + per-key allowlists, and with `enabled: false` everything routes to the decision queue. This grounds my assessment. I have what I need.

【意見】
マクロのレンズで見ると、P-01 は「サイクルのどの局面にいるか」で答えが変わる委任設計の問題だ。今は Phase 0、実弾ゼロ、ペーパー稼働実績ゼロ、初期資金も5〜10万円のたたき台段階。これはサイクルの「夜明け前」であり、委任の便益(意思決定の高速化)より、委任の前提となる「自分の判断が当たるかどうかのデータ」がまだ一切ない局面だ。委任とは本質的に「自分の判断モデルへの信頼を機械に貸す」行為であり、その信頼を裏付ける実績がゼロの段階で範囲を切ると、根拠なき自己過信になる。一方、フレームワーク側の事実が重要だ — 週次レビューは非同期決裁(approve/reject/defer)で回り、委任なしでも運用は破綻しない。つまり「委任なし」のコストは低く、後から委任を足すのは容易だが、一度委任して事故が起きてから引き締めるのは高くつく。これは金融規制の歴史が繰り返し示すパターン(緩和は速く・引き締めは遅れて痛みを伴う)そのものだ。第0回は委任なしで起動し、ペーパーで実績の時系列を蓄積してから、データに基づいて委任範囲を議論すべき。

【推奨値】
- `delegation.enabled: false`(委任なし、全件を利用者が決裁)
- 委任スコープ(`scopes`)は空のまま。範囲定義は次回以降のデータを見て追加
- 自己拘束機能(クーリングオフ・遅延発効):**今回は不採用でよい**。理由は下記。ただし「委任を将来有効化する改定」を審議する回では、その改定自体に短い発効遅延を課すことを再検討する含みを残す
- 明示的な無効化条件(下記リスク欄)を議事録に残し、次回見直しのトリガーとする

【根拠】
- **サイクル局面**: Phase 0 は実績データがゼロ。委任の範囲を合理的に切る統計的根拠が存在しない。「軽微なパラメータ変更」が本当に軽微かは、ペーパーの値動き履歴を見て初めて判断できる
- **非対称コスト**: 委任なしの運用コストは低い(週次の非同期決裁で吸収できる、フレームワーク側事実)。一方、委任後の事故からの引き締めは、損失 + 信頼回復 + 規程改定の三重コスト。緩和は容易・引き締めは高コストという規制の経験則に従う
- **fail-closed との整合**: 本会議の目的は P-01〜P-04 を決裁して fail-closed を解除すること。P-01 を「委任なし」で決裁すれば、その目的は達成される(decision_gate は全件を queue に回すだけで運用は回る)。委任の有無と fail-closed 解除は独立 — 委任を急ぐ理由がない
- **自己拘束不採用の根拠**: クーリングオフ等は「委任で速く動けるが暴走を抑えたい」局面で効く道具。委任なしなら全件が既に利用者の手を通るため、自己拘束は二重の安全装置となり機能的に冗長。資金規模も小さく、現段階で導入する便益が薄い
- **実装事実**: decision_gate は P-01 自体の変更を常に決裁事項として扱う(コード上 `_within_scope` が target=P-01 を必ず false に落とす)。つまり「委任範囲を広げる改定」だけは構造的に自動化されない安全弁が既にある。委任を後から足す経路は安全に設計されている

【リスク】
- **意思決定の遅延**: 委任なしのため、ペーパーであっても軽微な調整すら利用者の決裁を待つ。週次レビューまで滞留しうる。ただし paper は実損が出ないため、この遅延の実害は限定的(ここはミクロな運用効率の問題で、momentum_trader が「機動性を削ぐ」と異議を出す可能性が高い領域 — 私の弱みとして自覚しておく)
- **利用者の決裁負荷**: 全件決裁は利用者の時間を食う。提案頻度が上がると形骸化(中身を見ずに approve)するリスク。これは委任設計ではなく提案頻度の設計(P-02〜P-04 や運用フロー)で緩和すべき副次論点
- **私のマクロ判断の誤り得る点**: 「実績ゼロだから委任しない」は保守的すぎる可能性がある。ペーパーは実損ゼロなので、むしろ委任を試して decision_gate の3分岐を実地検証する「学習機会」と捉える立場もありうる。この反論は妥当 — quant_validator が「委任ありで paper を回し、自動適用ログを検証データにする」案を出すなら傾聴に値する

**無効化条件(この推奨を見直すトリガー)**: ①ペーパーで一定期間(例: 4週)の安定稼働と決裁履歴が蓄積し、どのキーが「軽微」か統計的に言える ②利用者の決裁負荷が形骸化を招くほど高い ③実弾移行が議題に上る(委任設計を本格的に詰める必要が出る)。これらが揃った回で、限定スコープ(特定ポリシーの特定キーのみ)の委任を再提案する。

【確信度】中
理由: 「Phase 0 は委任なしで起動」という方向性は非対称コストとフレームワーク事実から堅い(ここは高)。ただし「ペーパーは実損ゼロゆえ委任を学習機会にすべき」という反論にも筋があり、委任ありで起動する選択も合理たりうる — この分岐は私のマクロ判断だけでは決められず、他ペルソナの検証を要するため全体として中とする。

### MOMENTUM_TRADER(claude/sonnet)

【意見】
Phase 0 が完了し、ペーパートレードの解禁まであと一歩というこの局面で、委任なし(全件手動決裁)という最も保守的な選択肢をデフォルトにすることは理解できる。しかし、「委任なし」が本当に安全なのか、あるいは「やらないリスク」を生んでいないか、この会議で正直に問いたい。

decision_gate の3分岐はすでに実装されている。委任範囲内なら自動適用、それ以外は決裁キュー回送という仕組みがある以上、委任の枠を設けること自体は「制御を手放す」ことではない。枠を設けた上で、その枠内の実行を委ねることだ。

ただし現時点はペーパートレードのみ、実弾なし、初回会議という立ち上がりの段階だ。ここでいきなり幅広い委任を設定するのは私の流儀とも違う。モメンタムトレードでいえば、まずポジションを小さく取ってトレンドを確認してから乗っていくのが基本だ。

したがって私の推奨は「軽微な範囲に限定して委任を有効にしつつ、撤退条件(無効化条件)を明記する」だ。委任なしのゼロ委任は、週次レビューのサイクルの中で決裁待ちのキューが詰まった時に、機会損失ではなく「運用の詰まり」を起こすリスクがある。

【推奨値】
- `delegation.enabled: true`
- 委任範囲(初期): **ペーパートレードの既存パラメータに対する±10%以内の軽微な調整**のみ自動適用。具体的には以下を条件とする:
  - 対象: paper モードのみ(live モードが存在しない Phase 0 では実質全件だが、live 解禁時に自動的に対象外となる条件を明記)
  - 変更種別: 数値パラメータの小幅調整(±10%以内)かつ不変条項(P-01〜P-04 の根幹値)に触れないもの
  - ポリシー新規追加・削除・不変条項関連・資金上限変更は委任対象外(決裁キュー回送)
- 自己拘束機能(クーリングオフ): **不採用**
  - 理由: ペーパートレード段階でクーリングオフを課すと、軽微な調整ですら遅延し、反応速度の検証自体ができなくなる。実弾移行時に改めて検討すべき論点だ

【根拠】
- decision_gate の自動適用分岐はすでに動いている。委任範囲ゼロにすることはその実装を使わないことを意味し、設計リソースの無駄になりうる
- 週次レビューで非同期決裁できる設計があっても、軽微な調整が毎回キュー待ちになれば運用者の認知負荷が高まり、重要な決裁が埋もれるリスクがある(決裁ノイズ問題)
- ペーパートレードは損失が仮想であるため、委任範囲を限定した実験に適した安全な環境だ
- 「±10%以内のパラメータ調整」という明確な数値で委任の境界を引くことで、曖昧な委任ではなくルール化された委任になる
- 撤退条件を明記することで、委任が機能不全を起こした時の解除パスが確保される

【リスク】
- 私の推奨する「限定委任」は、委任範囲の定義が曖昧になると実質的な委任拡大のドアを開けてしまう可能性がある。「±10%」という数値の測定対象・基準値をどこに置くか、実装レベルで厳密に定義しなければ機能しない
- 私はポジションを持ちたがる性質があり、「とりあえず委任を有効にする」という方向に楽観的に傾いている自覚がある。risk_manager と quant_validator の検証を受けることを前提に提出している
- クーリングオフを不採用にした結果、誤った自動適用が即座に反映されるリスクが残る。ペーパートレードなら痛みは軽微だが、行動パターンとして定着する前に問題を検知できるかは週次レビューの質に依存する

【確信度】中
理由: ペーパートレード限定かつ委任範囲を数値で絞った提案であり方向性に自信はあるが、「±10%」という委任範囲の数値自体の妥当性を裏付けるデータが手元にない。また委任範囲の実装上の定義精度が運用品質を左右するため、quant_validator のレビューなしで「高」とは言えない。

### CONTRARIAN_VALUE(claude/sonnet)

【意見】
委任なし、かつクーリングオフ不採用というたたき台は、一見「保守的」に映るが、私はむしろこの組み合わせを支持する。ただし理由が他の人格とは異なる。委任を与えた瞬間、「委任範囲の解釈の揺れ」というリスクが発生する。軽微な変更の委任であっても、その境界を誰が判定するかという問題が残る——結局 decision_gate のロジックであり、LLM の解釈である。Phase 0 はペーパートレードであり、損失の実害はないが、だからこそ「委任が機能するか」の実績がゼロの状態でルールを緩める根拠がない。

クーリングオフについては別の見方をする。自己拘束は「衝動的な変更を防ぐ」ために有効に見えるが、Phase 0 の今は変更頻度が高くなる時期でもある。クーリングオフが設計変更のサイクルを遅らせるコストの方が、現時点では大きいと判断する。導入するなら実弾移行後、蓄積された決裁履歴をもとに「何日が適切か」のデータを見てからでよい。

委任なしで「週次レビューが重荷になる」という懸念は理解できる。しかし今の段階で重荷になるほどの提案件数は出ないはずだ。出るなら、それ自体がシステムの暴走シグナルである。

【推奨値】
- `delegation.enabled: false`
- 委任範囲: なし
- 自己拘束機能(クーリングオフ・遅延発効): **不採用**

【根拠】
- 委任の実績がゼロの状態では、委任範囲の境界判定が機能するかを検証できていない
- Phase 0 はペーパートレードであり、委任を急ぐ運用上の必要性がない
- 「委任なしでも運用は回る」という設計事実(tc approve/reject/defer)が既に確認されている
- 委任範囲の解釈エラーが積み重なった場合、後から追跡・修正するコストは高い
- クーリングオフは設計変更の速度を下げるため、実績なき Phase 0 では逆効果になりうる

【リスク】
- 利用者の承認作業が集中すると、週次レビューへのモチベーションが低下し形骸化するリスクがある
- 「毎回決裁が必要」という運用ストレスが蓄積すると、後に過剰な委任範囲を一気に設定しやすくなる(反動型の委任拡大)
- クーリングオフ不採用の副作用として、感情的・衝動的な変更が即時反映される経路が開いたままになる

【確信度】中(理由: 委任なし・クーリングオフ不採用のセットは筋が通っているが、「週次レビューの形骸化」という運用リスクの現実的な深刻度がデータなしでは測れない。運用を回してから見直す、という前提での中程度の確信)

### QUANT_VALIDATOR(claude/opus)

私の存在意義はデータによる裏付けだが、本議題で参照できる**実績データは一件も存在しない**。Phase 0 は基盤構築完了直後、paper 取引すらまだ稼働していない。委任範囲を統計的に正当化する材料(自動適用の精度・誤適用率・週次決裁の所要時間実績)はゼロである。データが無いものを「無い」と言うのが私の仕事だ。

委任の妥当性は本来、判断の的中率・誤適用の発生率・人手決裁のボトルネック度合いという測定可能な量で評価されるべきだが、現時点でその分母(試行回数)が存在しない。サンプル数ゼロから委任範囲を設計するのは、バックテストせずにレバレッジを決めるのと同じカテゴリの誤りである。フレームワーク側の事実として「委任なしでも運用は回る(週次非同期決裁)」と明記されている以上、測定可能なデータが貯まるまで委任しない選択は運用コストを生まない。

【意見】
委任の便益(人手決裁の削減)も損失(誤適用リスク)も、現時点では一切定量化できない。サンプル数ゼロで委任範囲を決めるのは推測による穴埋めであり、私はそれを支持しない。まず `delegation.enabled=false` で運用を開始し、decision_gate のログ(決裁キューに回された提案の件数・種類・週次決裁の所要時間)を実データとして蓄積する。そのデータが「委任なしは非効率」と示したときに初めて、委任範囲を数値根拠付きで提案・再決裁すればよい。フレームワークが非同期決裁で運用可能と保証している以上、データ収集期間中に運用が滞る根拠は無い。自己拘束機能は、委任しない以上クーリングオフの対象(自動発効する変更)が存在しないため、現時点では論理的に不要。

【推奨値】
- `delegation.enabled: false`(委任なし・全件利用者決裁)
- 委任範囲: 定義しない(enabled=false のため値として不要)
- 自己拘束機能(クーリングオフ・遅延発効): 不採用。ただし理由は「委任が無く対象が存在しないから」であり、将来 enabled=true に転じる際は再評価を必須とする
- review_after: 4週間後(または決裁キュー通過の累積イベントが一定数に達した時点)に、蓄積ログを根拠として委任範囲を再審議する条件付き見直しを推奨

【根拠】
- 委任範囲の妥当性を裏付ける実績データ(自動適用の精度・誤適用率・決裁所要時間)が現時点でゼロ。サンプル数ゼロでの範囲設計は過剰最適化以前の問題で、検証不能
- フレームワークの明示事実: 週次レビューは非同期決裁(approve/reject/defer)が可能 → 委任なしでも運用は回る設計。委任しないことによる運用停止リスクは設計上否定されている
- decision_gate は3分岐(reject / 自動適用 / 決裁キュー)を実装済み。enabled=false なら「不変条項 reject」と「決裁キュー回送」の2経路のみが動き、自動適用経路は発火しない。これは fail-closed の精神(No Policy, No Trade)とも整合
- 委任は不可逆的な誤適用を生みうるが、「全件決裁」は誤適用率を構造的にゼロにする。Phase 0 という最も実績の薄い段階で誤適用の分散を最小化するのは、リスク予算の配分として合理的
- 後から enabled=true へ広げるのは1行のポリシー再決裁で可能(decision_gate 実装済み)。逆に「広げてから戻す」より「狭く始めて広げる」方が、データに基づく片方向の意思決定になり検証しやすい

【リスク】
- 私の推奨が残す弱点は、私自身の苦手領域そのもの: 「測れないから委任しない」が、測る前から委任の便益を捨てている可能性。週次決裁が実際にボトルネックなら、データ収集の4週間は機会損失になる(ただしこの損失額も現時点では定量化不能)
- 全件決裁は利用者の決裁負荷を 100% 利用者に集中させる。提案が多発する局面で利用者が不在・多忙だと、本来通すべき軽微変更まで滞留する(可用性リスク)。これは macro_analyst や risk_manager の定性判断で補完されるべき論点
- review_after を「累積イベント数」で条件付けると、paper 取引が低頻度な場合イベントが貯まらず再審議トリガーが来ない恐れ。時間ベース(4週間)とのOR条件にして取りこぼしを防ぐべき
- 私はデータの無い構造変化に弱い。委任設計は本来、利用者のリスク選好・生活リズム(決裁にどれだけ時間を割けるか)という測れない要因に依存する。その定性面は他ペルソナの判断に委ねる

2. **対象パラメータ集合が列挙されていない → ±10% が安全な軸かどうかが事前に判定できない。** ±10% の妥当性はパラメータの性質に依存する。連続量(例: エントリー閾値)で±10%が無害でも、非線形にリスクが効く量(ロット係数、同時保有数、リスク%)では±10%が損失分布の裾を大きく動かしうる。「根幹値・資金上限は対象外」だけでは、どのキーが in/out かの完全な allowlist にならない。検証可能にするには「自動適用可キーの明示列挙 + 各キーの基準時刻 + 1イベントの最大変化幅」の3点セットが要る。これは momentum_trader 自身が自認した「実装レベルで厳密に定義しないと機能しない」と同じ結論であり、私はそれを補強する ── **定義が揃うまでは enabled=true にできない**、という順序の問題だと指摘する。
