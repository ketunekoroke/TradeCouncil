## 1. 背景と目的

### 1.1 背景
- 複数の売買BOTを **Claude Code を中心とした開発・改善ループ**で管理したい
- **人格(ペルソナ)を持つ複数のLLMエージェント**による戦略立案を行いたい
- **週次・月次**で各BOTの成績を評価し、**成績不振BOT(以下「悪BOT」)** を特定して改善・降格・退役させるフィードバックを回したい
- 情報収集(ニュース・市況)もBOT化し、**リアルタイムニュース起因の変動**からも収益機会を狙いたい
- 対象は単一市場に限定せず、**暗号資産・国内株・海外株・先物・債券系を同一基盤**で扱い、短期から長期まですべてのホライズンに対応したい(v0.2 で確定)

### 1.2 ゴール
| ID | ゴール |
|---|---|
| G-1 | 24時間無人で稼働する「情報収集 → 解析 → 売買 → 記録」パイプラインの構築 |
| G-2 | ペルソナエージェント群による戦略の継続的な生成・議論・配分決定の仕組み化 |
| G-3 | **決裁を経ない変更が構造的に不可能**で、全決定が監査可能な安全基盤(エージェントにも自動化にも迂回させない) |
| G-4 | 実績データ → エージェントによる分析・改善提案 → **利用者の決裁** → 反映、という改善ループの仕組み化 |
| G-5 | 利用者が**唯一の決裁権者**としてトップダウンで統制でき、エージェントは提案・審議までを担う**ガバナンス構造**。具体的な運用ルールはこの構造の上で「複数エージェント+利用者の会議」により継続的に策定・改定される |

### 1.3 非ゴール(明示的にやらないこと)
| ID | 非ゴール | 理由 |
|---|---|---|
| NG-1 | HFT(ミリ秒単位の速度競争)での優位獲得 | 個人環境・LLM経由では構造的に不可能。ニュースの「初動の一瞬」は機関・HFTが支配している |
| NG-2 | 利益の保証 | 本システムは収益を保証しない。期待値検証の仕組みを作るのが目的 |
| NG-3 | LLMによる注文の直接執行 | LLMは「提案」まで。執行・上限管理は決定的(deterministic)なコードが行う |
| NG-4 | 初期からの完全自動運用 | 週次レビューの承認は当面人間が行う(ゲート制で段階的に自動化検討) |

## 3. システム全体コンセプト

### 3.1 三層アーキテクチャ

```
┌──────────────────────────────────────────────────────────┐
│ L3 改善層(Claude Code)── 開発・週次/月次レビュー          │
│   ・実績DB/ログを読み、レポートと改善提案(diff)を生成      │
│   ・人間の承認後にコード/設定へ反映、バックテスト実行       │
│   ・対話セッション + ヘッドレス実行(claude -p)をcronで起動 │
└───────────────▲──────────────────────┬───────────────────┘
                │実績・ログ            │承認済み変更
┌───────────────┴──────────────────────▼───────────────────┐
│ L2 知能層(Claude API)── 提案のみ・執行権限なし            │
│   ・ニュース解析(3段フィルタ)                             │
│   ・ペルソナ戦略会議(週次/日次/臨時) → 決議JSON           │
└───────────────▲──────────────────────┬───────────────────┘
                │市況・KPI・ニュース    │検証済みシグナル/配分
┌───────────────┴──────────────────────▼───────────────────┐
│ L1 実行層(Python 常駐・決定的)── 24/7稼働                │
│   ・データ収集BOT(価格WS / ニュースRSS / 指標カレンダー)   │
│   ・売買BOT群(戦略ごとに独立)                             │
│   ・リスクガード(上限強制・サーキットブレーカ・キルスイッチ)│
│   ・DB / 通知 / 死活監視                                   │
└──────────────────────────────────────────────────────────┘
```

### 3.2 大原則(設計全体を貫くルール)
1. **LLM非執行原則**: お金を動かすコードパスにLLMの自由裁量を置かない。LLM出力は必ず「スキーマ検証 → ホワイトリスト検証 → 変化幅制限」を通過したものだけが config に反映される
2. **権限分離(三権分離)**: 提案権(エージェント・実績データ・利用者=誰でも)/ 審議権(ペルソナ会議体)/ **決裁権(利用者のみ)** を分離する。決裁を経ずに本番挙動が変わる経路を実装しない(FR-10)
3. **全判断に根拠ログ**: すべての注文は `decision_id` で「どのシグナル・どの決議・どの根拠文」に遡れる。これが週次レビューの一次資料になる
4. **ゲート制(ポリシー)**: バックテスト → ペーパートレード → 少額実弾 → 増額という段階構造はフレームワークが提供し、各ゲートの数値基準は会議+決裁でポリシーとして定める
5. **速度ではなく規律で戦う**: ニュースは「初動を取る」のではなく「数分〜数時間のドリフトを取る」「危険時にポジションを落とす」ために使う
6. **No Policy, No Trade(fail-closed)**: 損失上限・レバレッジ等の必須ポリシーが未決裁の領域では発注を拒否する。「ルールを今決めない」ことと「安全」はこの性質で両立する

## 4. 機能要件

### FR-1 情報収集BOT群
| ID | 要件 | 補足 |
|---|---|---|
| FR-1.1 | 市場データ収集: 対象銘柄の価格・出来高・板情報を WebSocket で常時受信し、1分足(必要に応じ秒足)をDBへ保存 | 切断時の自動再接続・欠損検知・RESTでの補完 |
| FR-1.2 | ニュース収集: RSS(取引所の告知、主要メディア)、経済指標カレンダーを定期ポーリング | X(Twitter)API は有料のため任意(Phase 2 以降に判断) |
| FR-1.3 | 正規化・重複排除: 同一ニュースの重複検知、受信タイムスタンプと公開タイムスタンプの両方を記録 | 「検知遅延」を後で測定するため |
| FR-1.4 | データ鮮度監視: 一定時間データが更新されない場合、警告通知+依存BOTの新規エントリー停止 | 古いデータでの誤発注防止 |

### FR-2 ニュース解析パイプライン(3段フィルタ)
| 段階 | 処理 | コスト/速度 |
|---|---|---|
| Stage 1 | ルールベース: キーワード辞書・銘柄辞書・ソース重み付けで足切り | 無料・ミリ秒 |
| Stage 2 | 軽量LLM(Haiku想定)で分類: `{影響度1-5, 対象銘柄, 方向(強気/弱気/不明), 確度, 影響の想定持続時間}` を固定JSONで出力 | 低コスト・数秒 |
| Stage 3 | 影響度4以上のみ上位モデルで詳細分析し、戦略会議への臨時アジェンダ or リスクオフ提案を生成 | 高コスト・低頻度 |

- レイテンシ目標: ニュース検知 → シグナル化まで **p50 < 30秒**(NG-1 の通り、速度競争用ではない)
- 出力は「売買シグナル」ではなく「**イベントシグナル**」としてDBに保存し、各BOTが自分のルールで参照する

### FR-3 ペルソナ戦略会議(マルチエージェント)
| ID | 要件 |
|---|---|
| FR-3.1 | 定例会議: **週次(深い議論)**・日次ブリーフ(軽量)・臨時(Stage 3 イベント発生時) |
| FR-3.2 | 初期ペルソナ5名: ①マクロアナリスト ②モメンタムトレーダー(順張り) ③逆張りバリュー ④クオンツ(データ・統計検証担当) ⑤リスクマネージャー(**拒否権あり**) |
| FR-3.3 | 会議プロトコル: 入力パッケージ(市況サマリ・KPI・ニュース要約)→ 各自意見(構造化JSON)→ 反論ラウンド1回 → 議長が決議JSONへ集約 → リスクマネージャー最終審査 |
| FR-3.4 | 決議JSONの内容: 相場観 / BOT別資金配分 / パラメータ変更案 / 新戦略候補 / リスク警告(イベント前のエクスポージャー削減指示等) |
| FR-3.5 | 決議は直接適用されない。L1 の検証層(スキーマ・ホワイトリスト・変化幅制限)を通過した項目のみ config に反映 |
| FR-3.6 | 全発言・全決議を `council_sessions` テーブルに保存(週次レビューで「誰の意見が当たったか」を採点) |

### FR-4 売買BOT群(実行層)
| ID | 要件 |
|---|---|
| FR-4.1 | 各BOTは独立した設定(strategy_id / 対象銘柄 / 資金配分上限 / パラメータ)を持ち、独立プロセスとして稼働・再起動可能 |
| FR-4.2 | 共通機能: ペーパー/実弾モード切替、注文の冪等性(二重発注防止)、再起動時のポジション復元、約定照合 |
| FR-4.3 | 初期戦略候補(Phase 1〜2 で各1本): ①トレンドフォロー ②レンジ逆張り ③ニュースドリフト(Stage 2/3 シグナル参照) ④イベント前リスクオフ(防御専用・収益ではなく損失回避が目的) |
| FR-4.4 | すべての発注に `decision_id` を付与し、根拠(シグナルID・決議ID)へ遡及可能にする |

### FR-5 リスク管理(決定的・最優先)
| ID | 要件 | 初期値の例 |
|---|---|---|
| FR-5.1 | 1取引あたり最大損失 | 口座残高の 0.5〜1% |
| FR-5.2 | 日次最大損失 → 当日全BOT新規停止 | 口座残高の 2% |
| FR-5.3 | 週次最大ドローダウン → 全停止+人間へ通知 | 口座残高の 5% |
| FR-5.4 | 最大同時ポジション数 / 銘柄別上限 / レバレッジ上限 | config で定義 |
| FR-5.5 | サーキットブレーカ: 急激な価格乖離・スプレッド異常・データ鮮度切れで新規エントリー停止 | — |
| FR-5.6 | キルスイッチ: コマンド・フラグファイル・チャット操作(Teams 等。Phase 0 は CLI/フラグのみ)のいずれからでも全BOT即時停止(+任意で全ポジション成行クローズ) | — |
| FR-5.7 | APIキーは**取引権限のみ(出金権限なし)**、可能ならIP制限を設定 | — |
| FR-5.8 | 資産クラス別レバレッジ上限を config で強制(法令上限と自主上限10倍の小さい方)。口座全体の**実効レバレッジ**にも別途上限を設ける | 初期: 全クラス1倍 → ゲート通過で段階解禁(基本設計書 §9) |
| FR-5.9 | 証拠金監視: 維持率がしきい値を割ったら新規停止→自動縮小。ブローカーの強制ロスカット水準より十分手前で自衛する | 例: 維持率300%で警告 / 200%で縮小開始 |
| FR-5.10 | オーバーナイト・ギャップ規則: 立会時間制の市場(株・先物)では引け前に実効レバレッジを上限の1/2へ自動縮小 | 窓開け(ギャップ)による想定外損失の抑制 |

### FR-6 記録・評価・フィードバック(週次/月次)
| ID | 要件 |
|---|---|
| FR-6.1 | 全注文・全シグナル・全決議・全インシデントを根拠付きでDBに保存 |
| FR-6.2 | **週次**: KPI自動集計 → Claude Code(ヘッドレス)がレビューレポート+改善提案(設定変更は diff 形式)を生成 → Teams に要約通知 → 人間承認後に反映 |
| FR-6.3 | **月次**: 戦略の昇格/降格/退役の判定、新戦略のバックテスト課題票を発行(Claude Code のタスクとして登録) |
| FR-6.4 | **悪BOT判定(初期基準)**: 「4週連続でKPI基準未達」または「最大DD超過」→ 自動で次の状態へ降格: `ACTIVE → REDUCED(配分半減) → PAPER(ペーパー降格) → RETIRED(退役)` |
| FR-6.5 | 好成績BOTの昇格は自動では行わず、月次レビューで人間が承認(過剰最適化への防波堤) |

### FR-7 監視・通知
| ID | 要件 |
|---|---|
| FR-7.1 | Microsoft Teams へ通知(Power Automate Workflows + Adaptive Card): 約定、損失警告(日次上限の50%/80%到達)、停止イベント、日次サマリ、週次レポート。severity 色と facts(キー値ペア)で構造化可視化する。Discord Webhook は backend 切替で予備(ADR-0002) |
| FR-7.2 | 死活監視: 各BOTの heartbeat をDBに記録し、途絶時に通知+自動再起動(systemd) |
| FR-7.3 | LLM APIコストメーター: 月額上限(A-6)の80%で警告、100%で知能層を自動縮退(Stage 2/3 停止、定例会議は週次のみに) |

### FR-8 マルチアセット基盤(v0.2 で追加)
| ID | 要件 |
|---|---|
| FR-8.1 | 統一インストゥルメントモデル: すべての銘柄を `instrument(資産クラス, 取引所, シンボル, 通貨, 呼値, 最小数量, 取引セッション, 証拠金規則)` として一元管理し、BOT・リスク管理・戦略会議は**資産クラス非依存のインターフェース**で動作する |
| FR-8.2 | ブローカーアダプタ: 発注・残高・建玉・約定照合を共通インターフェース化し、資産クラス/ブローカーごとに実装を差し替え可能にする(基本設計書 §2.5 マトリクス) |
| FR-8.3 | 市場カレンダー/セッション管理: 24/7(暗号資産)と立会時間制(株・先物)を同一スケジューラで扱う。取引所休日・SQ・配当落ち等のイベントカレンダーを管理し、場外時間の発注を抑止する |
| FR-8.4 | マルチ通貨: JPY/USD 等の口座残高を**円建て統合評価**し、PnL・リスク上限判定は統合評価額で行う(為替レートを定期取得) |
| FR-8.5 | 段階導入: 抽象化(FR-8.1〜8.4)は Phase 0 から導入し、市場の実装は縦切りで ①暗号資産 → ②IBKR(海外株・先物・債券ETF) → ③国内株API の順に追加する |

### FR-9 マルチタイムホライズン(v0.2 で追加)
| ID | 要件 |
|---|---|
| FR-9.1 | BOT実行型を2系統サポート: **(a) バー駆動型**(1分足〜日足のバー確定で判断。短期〜スイング向け) / **(b) スケジュール駆動型**(日次・週次・月次リバランス。中長期ポートフォリオ向け) |
| FR-9.2 | 戦略会議は **短期/中期/長期の3スリーブ**にまず資金を配分し、次にスリーブ内でBOT別配分を行う(ホライズン横断の一貫したリスク配分) |
| FR-9.3 | 長期スリーブは**恒久レバレッジ1倍(現物相当)**。レバレッジの段階解禁は短期スリーブのみを対象とする |
| FR-9.4 | KPI・悪BOT判定はホライズン別に評価期間を変える(短期=週次、長期=月次〜四半期。長期BOTを週次成績で退役させない) |

### FR-10 ガバナンス/ポリシー管理(v0.3 で追加 — 本フレームワークの中核)
| ID | 要件 |
|---|---|
| FR-10.1 | **ポリシーレジストリ**: すべての運用ルール(リスク上限・レバレッジ・ゲート基準・スリーブ配分・悪BOT判定・予算・委任範囲など)をバージョン管理された「ポリシー」として一元管理する。システムは「決裁済み(active)」のポリシーのみを読み込む |
| FR-10.2 | **決裁フロー**: 提案(エージェント・実績データ・利用者)→ 審議(ペルソナ会議)→ **決裁(利用者のみ)** → 適用 → 監査ログ。決裁レコードなしに本番反映される経路を持たない |
| FR-10.3 | **同期意思決定会議モード**: 利用者が議長として参加する対話形式の会議。ペルソナが審議し、利用者がその場で承認/修正承認/却下/差し戻しを宣言すると、決裁レコードとポリシー差分が自動生成される |
| FR-10.4 | **委任ルール**: 「この範囲の軽微な変更は自動適用してよい」という委任を、利用者が自身の決裁により定義・変更・撤回できる(初期値: 委任なし = 全件決裁) |
| FR-10.5 | **不変条項(フレームワーク憲法)**: ①決裁権は利用者のみ ②LLM非執行原則 ③全決定の監査ログ ④キルスイッチ ⑤No Policy, No Trade — これらは「ポリシー」ではなくフレームワークの一部であり、会議の議題にできない(基本設計書 §1.5.2) |
| FR-10.6 | **見直しサイクル**: 各ポリシーに review_after(見直し期限)を持たせ、期限到来時に自動で会議アジェンダへ再上程する |

## 8. リスクと免責(必読)

1. **自動売買は元本を失うリスクがある。** 本システムは利益を保証しない。個人の自動売買の多くは手数料とスリッページで期待値を失う。本設計はそれを「測定して淘汰する」ための仕組みである
2. **ニュースの初動はHFT・機関投資家が支配している。** 本設計は速度ではなく「持続的ドリフト・リスク回避・規律」で優位を探す前提に立つ(NG-1)
3. **LLMは誤る(幻覚・過信)。** よって執行とリスク上限から構造的に隔離する(NG-3、原則1)
4. 取引所の利用規約、API利用規約、税務(日本の暗号資産は原則雑所得・総合課税)、法令の確認は利用者の責任で行うこと。本書は投資助言ではない
5. **レバレッジは損失を線形に拡大する。** 10倍では10%の逆行で資金が消滅し、先物等のギャップでは預託金を超える損失(追証)もあり得る。よって本設計はレバレッジを「初期値」ではなく「**ゲートで段階解禁する上限**」として扱う(FR-5.8)
6. 日本の現行規制上のレバレッジ上限(2026年6月時点・実装時に最新を要確認): 国内暗号資産の証拠金取引は**個人2倍**、株式信用取引は**約3.3倍**、店頭FXは**個人25倍**。先物は証拠金制度による。つまり10倍級のレバレッジに実際に届くのは先物・FX・海外業者に限られ、海外業者の利用可否は利用者の判断と責任となる。なお暗号資産は金商法への移行(2027年度施行見込み)が進行中で、レバレッジ規制も見直される可能性がある
7. **個別債券は個人向けのAPI取引手段がほぼ存在しない。** 本基盤では債券エクスポージャーは債券ETF・金利先物で代替する
