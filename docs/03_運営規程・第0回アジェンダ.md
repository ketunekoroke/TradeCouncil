# 運営規程・第0回意思決定会議アジェンダ(TradeCouncil)

| 項目 | 内容 |
|---|---|
| バージョン | 0.3 |
| 位置づけ | **フレームワーク(不変)とポリシー(可変)を切り分け、ポリシーの決め方・変え方を定める文書**。基本設計書 §1.5 と対になる |
| 決裁権者 | 利用者(オーナー)ただ一人 |
| 本文書自体の改定 | 第1章(不変条項)以外は、本規程の決裁フローに従って改定できる |

---

## 第1章 不変条項(フレームワーク憲法)

以下はポリシーではなくフレームワークの一部であり、**会議の議題にできない**。

1. **決裁権は利用者のみ**に属する。決裁を経ずに本番挙動が変わる経路を実装しない
2. **LLM非執行原則**: エージェントは提案・審議まで。執行と強制は決定的コードが行う
3. **全決定の監査ログ**: 提案 → 審議 → 決裁 → 適用 → 結果を欠落なく記録する
4. **キルスイッチ**: 利用者はいつでも全停止できる
5. **No Policy, No Trade(fail-closed)**: 必須ポリシー(★印)が active でない領域では発注を拒否する

## 第2章 役割と決裁フロー

### 2.1 役割

| 役割 | 担い手 | 権限 |
|---|---|---|
| 決裁権者 | 利用者 | 承認 / 修正承認 / 却下 / 会議差し戻し / 期限付き保留。委任の設定・撤回 |
| 議長(facilitator) | エージェント | 会議進行、論点整理、決議案の起草。**決定はしない** |
| ペルソナ(5名) | エージェント | 審議・意見・反論。risk_manager は veto(審議段階の差し戻し)を持つが、これは決裁の代替ではない |
| 起案者 | エージェント / 実績データ(自動起案) / 利用者 | 何でも提案できる(リスク上限・レバレッジ含む) |
| 書記・実装 | Claude Code | 議事録・決裁レコードの生成、決裁済みポリシーの実装反映 |

### 2.2 標準フロー

```
起案(proposals/) → 審議(会議 or 週次レビュー) → 決裁(利用者) → 適用(config/policies/ 更新+実行用ビュー再生成)
   → 監査(policy_decisions に記録) → 見直し(review_after 到来で自動再上程)
```

### 2.3 決裁レコード必須項目

policy_id / version / 決裁内容(承認・修正承認・却下・保留)/ 根拠参照(会議ID・データ)/ チャネル / 日時。
決裁レコードのないポリシー変更はシステムが受理しない(pre-commit と decision_gate の双方で検査)。

## 第3章 会議体規程

| 会議 | 頻度 | 形式 | 主な内容 |
|---|---|---|---|
| 第0回(キックオフ) | 1回 | 同期 | 初期ポリシー一式の決裁(第5章アジェンダ) |
| 週次定例 | 毎週 | 非同期可 | 実績レビュー、改善提案の決裁、悪BOT状態遷移の確認 |
| 月次戦略会議 | 毎月 | 同期推奨 | スリーブ配分、戦略の昇格/退役、review_after 到来ポリシーの見直し |
| 臨時 | 随時 | どちらでも | 重大ニュース(Stage 3)、インシデント、利用者の発議 |

### 同期会議の式次第(議長が進行)
1. 会議パッケージの提示(市況サマリ / KPI / 審議対象の提案一覧)
2. ペルソナ意見(Round 1: 並列)→ 相互反論(Round 2)
3. **利用者の質疑**(自由に深掘り・追加シナリオの検討を指示できる)
4. 議長が選択肢を「案A/案B/保留」の形に確定し、各案の根拠とリスクを要約
5. 利用者が決裁を宣言(議題ごと)
6. 書記が決裁レコードを読み上げ、利用者の最終確認をもって確定

議事録は council_sessions に全文保存し、決裁事項のみ policy_decisions へ転記する。

## 第4章 ポリシーレジストリ運用

- ライフサイクル: `draft → proposed → approved → active → retired`(形式は基本設計書 §1.5.3)
- どんなポリシーも**提案は自由、適用は決裁のみ**。ロールバックは旧バージョンの再決裁(履歴は不滅)
- 実行用 config(risk_limits.yaml 等)は active ポリシーからの**自動生成ビュー**。手編集は禁止・検出対象
- 委任: 利用者は P-01 により「自動適用してよい範囲」を定義できる。委任範囲の変更そのものは常に決裁事項

## 第5章 第0回会議アジェンダ(たたき台つき)

> 「たたき台」列は設計時(v0.2)の検討値。**そのまま採用しても、会議で変えても、保留してもよい。**
> ★ = 必須ポリシー。これが決裁されるまで該当領域は fail-closed(取引不可)。

| ID | 議題 | たたき台(参考値) | 会議での主な論点 |
|---|---|---|---|
| P-01 ★ | 決裁・委任規程 | 委任なし(全件決裁)。自己拘束機能(クーリングオフ等)は不採用 | どこまで自動適用を許すか / 大変更に自己拘束を課すか |
| P-02 ★ | レバレッジ規程 | 全クラス1倍で開始、8週ゲートで段階解禁、上限 = min(法令, 10倍)、長期スリーブ1倍固定 | 開始倍率 / 解禁条件 / スリーブ別の扱い / 海外業者の利用可否 |
| P-03 ★ | 口座リスク上限 | 日次 -2% で当日停止 / 週次DD -5% で全停止 / 1取引 -1% / 総エクスポージャー60% | 絶対値 / 停止後の再開手続き |
| P-04 ★ | セーフガード運用 | 維持率300%警告・200%縮小 / 引け前に実効レバ50%へ / CB: 1分5%変動・スプレッド30bps・データ鮮度90秒 | しきい値 / 対象市場ごとの差 |
| P-05 | 悪BOT判定・淘汰規程 | 4週連続KPI未達 or DD超過 → REDUCED → PAPER → RETIRED。評価期間はスリーブ別 | 基準KPI / 連続週数 / 復帰条件 |
| P-06 | フェーズゲート基準 | バックテスト: PF>1.2・DD<15%・100取引以上 / ペーパー8週 → 実弾 / 実弾12週 → 増額 | 期間 / 指標 / 例外手続き |
| P-07 | スリーブ初期配分 | 短期40 / 中期40 / 長期20(%) | 配分比率 / 見直し頻度 |
| P-08 | 予算規程 | 月1万円上限。80%で警告、100%で知能層を縮退 | 上限額 / 縮退の優先順位 |
| P-09 | 会議サイクル | 週次非同期 + 月次同期 + 臨時 | 頻度 / 利用者の所要時間 |
| P-10 | 初期資金・増額規程 | 5〜10万円で開始 | 金額 / P-06 ゲートとの連動 |
| P-11 | 通知・エスカレーション | Microsoft Teams(Power Automate Workflows 経由・Adaptive Card — ADR-0002)。severity 別に即時/日次まとめを分ける。Discord は backend 切替で予備 | 何を即時に知りたいか / 予備チャネルの要否 / Notion ミラー(docs/proposals/2026-06_notion-mirror-proposal.md)の採否 |
| P-12 | ペルソナ編成 | 初期5名(基本設計書 §5.1) | 追加・入替の手続き / 採点による改廃 |

> 第0回で全部を決める必要はない。★の4件だけ決裁すれば Phase 0 のペーパー稼働は開始でき、残りは週次・月次で順次決裁すればよい。

## 第6章 第0回の開催方法(2通り)

**A) Claude Code 上で開催(推奨・Phase 0 終盤)**
リポジトリ整備後、`/council 第0回` を実行。本章のアジェンダが読み込まれ、ペルソナが審議、利用者の決裁宣言で config/policies/ に決裁済みポリシーが生成される。

**B) 先行開催(今すぐ可能)**
claude.ai の新しい会話に本書と基本設計書を添付し、次のように指示する:
「03 の第5章アジェンダで第0回意思決定会議を開催してください。あなたが議長としてペルソナ5名(基本設計書 §5.1)を演じ分け、私が決裁権者です。議題は P-01 から1件ずつ、式次第(第3章)に従って進めてください。」
→ 決裁結果(各ポリシーの確定値)を持って Phase 0 の実装に入れば、Claude Code は最初から正式ポリシーで config/policies/ を構築できる。
