## 1. 開発フロー標準

### 1.1 1サイクルの定義 — BACKLOG アイテム単位の6ステップ

開発は常に **BACKLOG.md の1アイテム(BL-NNN)= 1サイクル**で回す。大きな構想も
BL に分割してから着手する(アジャイル・小刻み)。

| # | ステップ | 内容 |
|---|---|---|
| ① | BL 選択 | BACKLOG のアイテムを「今スプリント」へ移動。受け入れ条件を確認 |
| ② | docs 先行更新 | **ドキュメント駆動**: 仕様・設計の変更は docs/01〜05 の改訂を実装に先行させる。大きな判断は ADR に記録 |
| ③ | テスト先行 | 受け入れ条件をテストに翻訳して先に書く(red を確認) |
| ④ | 実装 | テストが green になるまで。`python -m scripts.cli test` 全緑が完了条件 |
| ⑤ | 管理表同期 | DOCS.md → REQUIREMENTS.md → FEATURES.md → TESTCASES.md |
| ⑥ | コミット | Conventional Commits で機能単位に分割。BL を「完了」へ移動 |

### 1.2 テストファースト原則

- **全モジュールで原則テスト先行**(③)。とくに `core/risk/`・`core/execution/` は
  **必須**(CLAUDE.md 絶対ルール5、risk はカバレッジ90%ゲート)
- 新機能の受け入れ条件は「pytest で機械検証できる形」に落とす。できないもの
  (シナリオ・人格)は TESTCASES.md の P1(手動)に登録する

### 1.3 ドキュメント駆動

- 一次資料(docs/01〜05)は**常に現状を映すよう直接改訂**する。改訂が実装に先行する
- 「コードを読まないと仕様が分からない」状態を作らない — これは人間のためであると
  同時に、**AI のコンテキストを節約する**ためでもある(§2.2)

### 1.4 自動化による下支え(設定済み)

| フック | 役割 |
|---|---|
| PostToolUse: post_edit_test.py | core/bots/tests の編集後に高速テストを自動実行 |
| PreToolUse: block_live.py | 実弾系コマンド・キル解除を AI からブロック |
| PreToolUse: protect_paths.py | policies/generated/var/prototype の手編集をブロック |
| git pre-commit | 秘密・決裁レコードなしポリシー変更を検出 |

## 2. マイクロ化と LLM コスト統制

### 2.1 作業単位の上限

- **1 BL = 1 モジュール境界**(core/notify、core/risk、scenarios/ 等)を原則とする。
  複数モジュールにまたがる場合は BL を分割する
- 効果: 1サイクルで AI が読むコンテキストがモジュール+関連 docs に閉じ、
  トークン消費が作業規模に比例しなくなる

### 2.2 コンテキスト衛生

- 着手時に読むのは「対象モジュール + 該当する docs の節 + 管理表の該当行」のみ。
  リポジトリ全体の再探索を毎回しない(docs が正確であることが前提 — §1.3 と表裏)
- 設計判断が必要な作業はプランモード(調査→計画→承認→実装)を使い、
  実装フェーズの手戻り(=コンテキスト浪費)を防ぐ

### 2.3 モデルの使い分け

- ペルソナは frontmatter(backend/model)で安いモデルに切替可能(haiku/sonnet/gpt-4o 等)。
  定型的な役割ほど安いモデルへ
- 定型ジョブ(週次レビュー等、Phase 4)は `claude -p`(ヘッドレス)+ 固定プロンプトで実行し、
  対話セッションを使わない

### 2.4 コスト計測と予算

- LLM 呼び出しは llm_usage テーブル(Phase 2 実装)に記録し、P-08(予算規程)と
  A-6(月額上限1万円)に接続する。80% で警告、100% で知能層を縮退(FR-7.3)

### 2.5 マイクロ化の発動条件

以下のいずれかが恒常化したら、BL を起票して運用を一段細かくする
(モジュール README の整備、BL の粒度半減、定型作業の claude -p 化など):

- 1サイクルの開始時に読む必要があるファイルがモジュール外へ恒常的に広がる
- テストスイートの実行時間が編集後フックのタイムアウト(360秒)に近づく
- 月次 LLM コストが A-6 の 50% を超える

## 3. 環境モデル

### 3.1 3つの環境

| 環境 | 実体 | 用途 | 寿命 |
|---|---|---|---|
| **dev** | 手元 Windows(本リポジトリ + `var/`) | 開発・テスト・Phase 0 の paper 運用 | 常設 |
| **sandbox** | `TC_VAR_DIR` で差し替えた `var-<name>/` | 結合確認・実験(**作って壊す**) | 使い捨て |
| **prod** | サーバ 1台(Phase 1 以降。§4 で決定) | paper 常駐 →(Phase 5)live + paper 同居 | 常設 |

### 3.2 テスト環境は常設しない(本方針の核)

通常の「テスト環境/本番環境」の二重常設はしない。理由:

1. **単体/E2E テスト**は pytest が一時ディレクトリに毎回 DB を作って壊す(実装済み)— コストゼロ
2. **結合確認**は TC_VAR_DIR サンドボックス(§3.3)で数秒で作れて、ディレクトリ削除で消える。
   SQLite + ファイルベース構成なので「環境」の実体はディレクトリ1つであり、
   作って壊すコストはほぼゼロ
3. **paper 運用が恒久ステージング**: paper BOT は本番と同一コード・同一ポリシー・同一経路
   (risk_guard → executor)で常時走る。悪BOT降格(ACTIVE→REDUCED→PAPER)の仕組みにより、
   Phase 5 以降も paper は live と並走し続ける — 専用ステージングは構造的に冗長

「本番のみで開発」も採らない: 開発作業ツリーと運用環境の分離は git が担い(§5)、
動作確認はサンドボックスが担う。本番(prod)に未コミットの変更を置かない。

### 3.3 TC_VAR_DIR サンドボックス

実行時生成物一式(DB・キルフラグ・ログ = `var/` 配下)を環境変数 `TC_VAR_DIR` で
差し替える。**config/(ポリシー含む)と `.env` は共有** — fail-closed の状態や決裁済み
ポリシーが本番と一致したまま検証できるのは意図した仕様。

```powershell
# 1) サンドボックス指定(このコンソールのプロセスのみ。起動前に確定させること)
$env:TC_VAR_DIR = "var-sandbox"     # 相対 → <repo>/var-sandbox/。絶対パスも可

# 2) 別 DB を初期化して並走
.venv\Scripts\python.exe -m scripts.cli db init
.venv\Scripts\python.exe -m scripts.cli paper --bot dummy_rw

# 3) 検証(別コンソールでも同じ TC_VAR_DIR を設定してから)
.venv\Scripts\python.exe -m scripts.cli status   # 表示パスで対象環境を必ず確認
.venv\Scripts\python.exe -m scripts.cli kill     # → var-sandbox/run/KILL(本体とは独立)

# 4) 後片付け(ディレクトリごと破棄)
Remove-Item Env:TC_VAR_DIR
Remove-Item -Recurse -Force var-sandbox
```

ルール:

- 命名規約は **`var-<name>`**(repo 直下)。gitignore(`/var-*/`)と hooks の手編集保護の対象
- `TC_VAR_DIR` は**プロセス起動前に確定**させる(DB エンジンはプロセス内でキャッシュ
  されるため途中切替は反映されない)
- シェルに環境変数が残留すると以後の CLI が別環境を向く。`status` のパス表示で
  対象環境を確認する習慣をつける(テストは conftest が自動で隔離する)

### 3.4 実弾同居の防御層(Phase 5)

live BOT と paper BOT は**同一サーバに同居**する(ADR-0004)。物理分離の代わりに
ソフトウェアの多層防御で守る:

| 層 | 防御 |
|---|---|
| 取引所 API キー | **出金権限なし**のキーのみ使用(FR-5.7) |
| 経路 | 全注文が risk_guard を通る(型レベル遮断)+ fail-closed(No Policy, No Trade) |
| AI 操作 | hooks(block_live.py)が実弾系コマンド・キル解除を AI からブロック |
| 緊急停止 | キルスイッチ(`var/run/KILL`)。解除は人間専用 |
| 監査 | 全注文 decision_id 遡及 + `tc kpi` の orphan ゼロ検証 |

分離の再検討トリガー: 実弾資金が一定額を超える、または live 系のインシデントが発生した
場合は、live 専用サーバへの分離を会議に付議する(判断基準を ADR-0004 に記録)。

### 3.5 AI のアクセスモデル(LLM 非執行原則の環境版)

「AI が本番とテスト両方の実態を見る必要がある」という要請には、
**観測と変更の非対称**で応える:

- **観測(read)= 直接可**: AI は本番の DB・ログ・`tc status`/`tc kpi` を直接読んでよい。
  Teams 通知も観測チャネルの一部
- **変更(write)= 必ず git + 決裁ゲート経由**: 本番での直接編集・直接コマンド変更は禁止。
  コード変更は手元で開発 → テスト → git push → デプロイ(§5)。
  運用ルール変更は `tc policy record`(決裁レコード)のみ

これにより「AI が実態を把握できない」問題と「AI が本番を壊す」リスクを同時に解消する。

### 3.5.1 本番データの閲覧経路(データ種別 × 経路 — ADR-0005)

**大原則: git は一方向(開発機 → push → 本番 pull = デプロイ)。本番は push しない。**
本番で生成されるデータは git を通さず SSH / snapshot / SharePoint で開発機へ還流する
(本番に push 認証を置かない・作業ツリーを乖離させない)。

| データ | 生成場所 | 開発機からの経路 |
|---|---|---|
| ポリシー / 決裁レコード | 開発機(決裁) | **git**(config/policies/ は追跡済) |
| 会議議事録(.md) | **開発機**(council 会議) | **git**(gitignore 例外で `council/*.md` を追跡) |
| シナリオ成果物(review/deliberation/メディア) | 開発機 or 本番 | **SharePoint 同期**(`scripts/sharepoint.py pull/push`) |
| DB トランザクション(注文/KPI/建玉/決裁履歴) | 本番 | **SSH ライブ読取**(下記) |
| DB 整合スナップショット / バックアップ | 本番 | **`tc snapshot`** → SharePoint/scp |
| 週次レビュー/提案(Phase 4) | 本番 | SharePoint/snapshot。採否は開発機で決裁→git |
| 可視化(議事録/ポリシー/KPI) | — | **Notion ミラー**(閲覧専用・MCP 一方向) |

SSH ライブ読取の例(DB ファイルを動かさず、本番で読取コマンドを実行して出力だけ受ける):

```powershell
ssh prod "cd /opt/tradecouncil && .venv/bin/python -m scripts.cli kpi"
ssh prod "cd /opt/tradecouncil && .venv/bin/python -m scripts.cli status"
```

**生 SQLite を WAL 稼働中のまま直接コピー/共有フォルダ越しにオープンしない**
(チェックポイント前の WAL を巻き込み破損・不整合の恐れ)。整合コピーが要るときは
`tc snapshot`(`VACUUM INTO`)を使う。詳細手順は docs/setup/remote-data-access.md。

## 4. ホスティング選択肢と判断基準(Phase 1 で決定 = BL-016)

A-5(稼働環境)は **Phase 1 着手時に本表を最新化して決定**する(ADR-0004)。
決定したら docs/01 A-5 を確定し ADR を起票する。

| 観点 | 国内 VPS(東京) | Azure VM | 自宅常時稼働機 |
|---|---|---|---|
| 月額コスト | ~700〜1,500円(2GB級) | ~2,000〜4,000円(B2級) | ほぼ0(電気代のみ) |
| A-6(月1万円)内の比重 | 小 — LLM 予算を圧迫しない | 中 | 最小 |
| 可用性(99%目標) | ◎ データセンター | ◎ 同左 + SLA | △ 停電・回線・Windows Update |
| 運用負荷 | 低(Linux + systemd、docs/02 既定構成) | 低〜中(Entra/課金管理が加わる) | 中(物理管理が人間に乗る) |
| 組織ガバナンス | 独立 | ◎ 既存 M365 テナントに統合(監査・ID) | 独立 |
| レイテンシ | ◎ 東京リージョン | ◎ Japan East | 回線次第 |

判断の目安: コスト最優先なら VPS、組織の Azure 管理に乗せたいなら Azure VM。
自宅機は 24h 試験等の暫定用途に限る(可用性99%目標と相性が悪い)。

## 5. デプロイフロー(Phase 1 以降)

### 5.1 標準フロー

```
手元で開発(§1 の6ステップ)→ git push
→ サーバ: git pull → 依存更新(pip install -e .)→ systemd restart
```

- **git は一方向**: 本番サーバは `git pull`(デプロイ)のみで、**push しない**(ADR-0005)。
  本番に push 認証を置かず、作業ツリーをデプロイ済みコミットと一致させる
- 本番で生成されるデータ(DB・週次レポート等)は git を通さず、SSH ライブ読取 /
  `tc snapshot` / SharePoint で開発機へ還流する(§3.5.1)
- BOT 再起動時は executor.reconcile() が建玉を突合する(実装済み)ため、
  paper の中断・再開は安全
- デプロイは人間が実行する(Phase 4 で自動化する場合も決裁ゲートを経る)

### 5.2 ロールバック

- `git revert` → 再デプロイ。DB スキーマは前方互換を保つ(docs/04 §5。
  破壊的変更は Alembic 導入後 = BL-010)

### 5.3 サーバ上の Claude Code 運用ルール

サーバには Claude Code を同居させる(BL-017 で整備)が、用途を限定する:

| 用途 | 可否 |
|---|---|
| `claude -p` の定型ジョブ(週次レビュー等、Phase 4) | ○(読み取り + reports/proposals 配下への書き込みのみ) |
| 障害調査の SSH 対話セッション(DB・ログの読解、原因特定) | ○(観測のみ) |
| 本番作業ツリーの直接編集・コミット | **禁止**(修正は手元 → git → デプロイ) |

hooks(block_live / protect_paths)はサーバ上でも同一設定で効かせる。
