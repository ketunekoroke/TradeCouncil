## 背景

Phase 1 以降、本番(paper/live BOT)は別サーバで常駐し、開発は手元 Windows 機で行う
(ADR-0004)。開発機から本番の **議事録・DB データ・ファイル** を閲覧したい。
docs/05 §3.5 は「観測=read は直接可 / 変更=git+決裁ゲート経由」を定めているが、
その「観測」の具体的経路がデータ種別ごとに未確定だった。

検討中、「**本番サーバが git commit/push するのは問題ではないか**」という指摘があった。
本番に push 認証を置くのはセキュリティ面の拡大であり、本番の作業ツリーが
「デプロイ済みコミット + 未コミット生成物」で乖離する(docs/05「本番に未コミット変更を
置かない」に反する)。この懸念への回答も本 ADR で確定する。

## 決定事項

### 1. git は一方向(開発機 → push → 本番 pull = デプロイ)。本番は push しない

- 本番サーバに push 認証を置かない。本番は `git pull`(デプロイ)のみ
- これにより本番の作業ツリーは常にデプロイ済みコミットと一致する

### 2. コミットを生む作業(会議・決裁・議事録生成)は開発機で行う

- council 会議・合議等のシナリオ実行(Claude Code ファシリテーター)は開発機側の作業。
  よって議事録の生成・コミットは開発機で完結し、本番 push は発生しない
- ポリシー決裁(`tc policy record`)も開発機で行い、git push → 本番 pull で反映する

### 3. 観測(read)の経路をデータ種別で確定する

| データ | 生成場所 | 経路 |
|---|---|---|
| ポリシー / 決裁レコード | 開発機 | git(config/policies/ は追跡済) |
| 会議議事録(.md) | 開発機 | git(本 ADR で gitignore 例外として追跡) |
| シナリオ成果物(review/deliberation/メディア) | 開発機 or 本番 | SharePoint 同期(既存 scripts/sharepoint.py) |
| DB トランザクション(注文/KPI/建玉/決裁履歴) | 本番 | **SSH ライブ読取**(本番で `tc status/kpi/policy list` を実行し出力を受領) |
| DB 整合スナップショット / バックアップ | 本番 | **`tc snapshot`**(VACUUM INTO)→ SharePoint/scp で配布 |
| 週次レビュー/提案(Phase 4 claude -p) | 本番 | SharePoint/snapshot で還流。採否は開発機で決裁→git |

- **生 SQLite を WAL 稼働中のまま直接コピー/共有フォルダ越しにオープンしない**
  (チェックポイント前の WAL を巻き込み破損・不整合の恐れ)。
  整合性のあるコピーが必要なら `tc snapshot`(`VACUUM INTO`)を使う

### 4. Notion ミラーを採用する(閲覧専用)

- 議事録・ポリシー台帳・週次 KPI 等を Notion へ一方向ミラーする運用を採用する
  (docs/proposals/2026-06_notion-mirror-proposal.md)。コードは書かず、接続済み
  Notion MCP をファシリテーターがシナリオ事後処理で使う
- これは Teams(ADR-0002)と同じ**ツール採用**であり fail-closed ポリシーではない。
  通知・可視化方針としての**正式批准は第0回会議の P-11 決裁**で行う

## 却下した代替案

| 案 | 却下理由 |
|---|---|
| 本番から git push(議事録・レポートを本番がコミット) | push 認証の本番配置・作業ツリー乖離。§1/§2 で回避 |
| 生 DB ファイルを共有フォルダ/rsync で同期 | WAL 稼働中のコピーは破損・不整合の恐れ。`tc snapshot` で代替 |
| 本番で直接編集して反映 | LLM 非執行原則・監査性に反する(docs/05 §3.5) |
