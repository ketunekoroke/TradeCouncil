| 項目 | 内容 |
|---|---|

| 関連 | docs/02 §12(課題票)§14(スラッシュコマンド)/ docs/06_戦略開発ガイド.md / docs/strategies/README.md / FR-6.3 / P-06(ゲート基準) |

## 背景

BOT 戦略の枠組み(`bots/base.py` の Strategy ABC + `STRATEGIES` レジストリ +
`config/bots/*.yaml`)は Phase 0 で整備済みだが、戦略は試験用ダミー1本のみで、

1. **アルゴリズムのノウハウ**(仮説・ロジックの意図・パラメータの根拠・検証結果・
   運用での学び)を記録する場所がない。ADR は技術アーキテクチャ判断、BACKLOG は
   タスク管理であり、戦略知識の置き場ではない
2. 新しい BOT を作る**標準手順・テンプレートがない**(ダミー戦略の見様見真似のみ)

docs/02 には既に「新戦略は課題票(hypothesis/検証データ/合格基準)として発行」
(§12)と `/new-strategy <name>`(雛形生成、§14)の仕様言及があり、本 ADR は
その具体化として蓄積の器と生成手段を確定する。

## 決定事項

### 1. 戦略ノウハウの真実源は `docs/strategies/` の戦略カード(1戦略=1ファイル)

- カードの書式・運用ルールは `docs/strategies/README.md` と `_template.md` が定める
- 状態は `draft → backtest → paper → live → retired`(P-06 のゲート段階と対応)
- 「運用での学び」セクションは日付付き **append-only**(失敗の記録こそ資産。
  消さない・上書きしない)
- **実績数値は DB が真実源**(`tc kpi` / bot_kpi_weekly)。カードには数値を転記せず、
  評価の解釈・学び・次のアクションを書く(二重管理を作らない)
- カードは会議(council / deliberation)でペルソナが引用できる知識源とする。
  秘密情報(API キー・残高等)は書かない(議事録と同じ作法)

### 2. 課題票の発行先を docs/adr/ から docs/strategies/ に変更する

- docs/02 §12 の「採択された新戦略は課題票として docs/adr/ に発行」を改め、
  **戦略カード(status: draft)として docs/strategies/ に発行**する。
  課題票(hypothesis / 検証データ / 合格基準)はカードの初期セクションそのものであり、
  検証・運用を経て同じファイルが育つ(発行文書と知識蓄積文書を分けない)
- ADR は技術アーキテクチャ判断に限定する(本来の用途に戻す)

### 3. 雛形生成は `tc bot new <bot_id> --strategy <key>`(CLI)

- 戦略ファイル(bots/)・BOT 設定(config/bots/、**enabled: false 既定**)・
  テスト雛形(tests/bots/)・戦略カード(docs/strategies/)の4点を一括生成する
- 既存ファイルがあれば**何も書かずに拒否**(部分生成しない)
- カードは `_template.md` から生成する(テンプレートの二重持ちをしない)
- docs/02 §14 の `/new-strategy` は本 CLI の薄いラッパとして将来実装する

### 4. `STRATEGIES` レジストリへの登録は自動編集しない(テスト駆動で誘導)

- スキャフォールドは `bots/__init__.py` を書き換えない。生成されるテスト雛形に
  「`STRATEGIES["<key>"]` が登録済みか」の検査を含め、生成直後は **red** になる
  → 開発者がレジストリ登録して green にする(テストファースト原則の具体化)
- 自動編集を避ける理由: レジストリは「どの戦略が実行可能か」を決める安全上の
  境界であり、人間のレビューを伴うコード変更として扱う

## 却下した代替案

| 代替案 | 却下理由 |
|---|---|
| DB 中心管理(戦略メタデータ・学びをテーブル化) | 学び・仮説は長文の知識であり、ドキュメント駆動文化(docs が真実源)と git レビューに乗せる方が蓄積・引用しやすい。実績数値は既に DB にあり、知識と数値の役割分担で足りる |
| Notion に直書き | Notion は閲覧専用ミラー(ADR-0005)。真実源を Notion に置くと git 管理・会議での引用・AI の参照が二重化する |
| ADR を課題票として流用(docs/02 原案) | ADR は不変の判断記録で追記運用に向かない。戦略カードは検証・運用で育つ生きた文書であり性質が異なる |
| レジストリ自動登録 | 実行可能な戦略の追加が無レビューで起きる。コード変更は人間の確認を経る原則を維持 |
