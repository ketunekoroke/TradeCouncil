# ロードマップ(段階導入案)

| 項目 | 内容 |
|---|---|
| 位置づけ | 構想(アイデア)。docs/ の一次仕様ではなく、TradeCouncil 本体の仕様・ポリシー・挙動を一切変更しない |
| 効力 | 非規範。フェーズ区切り・ゲート基準はすべてたたき台であり、決裁されたものは存在しない |
| 流用元 | TradeCouncil Phase 0 の思想(`docs/02_基本設計書.md` §9、`docs/05_開発フロー・実行環境方針.md`) |

## 1. 導入の思想 — ポリシー決裁が先、実行は後

TradeCouncil Phase 0 の3原則をそのまま導入の背骨にする:

1. **第0回会議がすべてに先行する**: ★SP-C 一式が決裁されるまで送信支援はゼロ
   (fail-closed)。ツールが先・ルールが後、の順番を許さない
2. **たたき台値を既定値にしない**: 本構想に登場する数値(値引き委任 10% 等)はすべて
   例示であり、確定値は会議の決裁でのみ生まれる
3. **フェーズゲートは測れる基準+決裁**: 各フェーズに定量基準を置き、基準を満たしても
   **自動では次へ進まない**(進めるかどうか自体が決裁事項)

## 2. リスク勾配 — 何から自動化し、何を最後にするか

対外到達のリスクが小さい順に進める:

```
社内向け(送信ゼロ) → ドラフトのみ → 単一顧客に実送信 → 多顧客展開 → 一斉配信
```

- 原案の「まずメール文面作成から」はこの勾配と整合する(ドラフトのみ → 実送信)
- **対案(併走)**: 経営会議(UC-4)は社外送信ゼロでリスク勾配の最下段にあり、メール支援の
  実装を待たずに「KPI 見える化+多視点審議+ポリシー決裁」のループだけ先に回せる。
  早期から並走させるかは Q-07([05_リスク・論点](05_リスク・論点.md) §9)

## 3. フェーズ計画

| Phase | ゴール | 主な内容 | ゲート(次へ進む条件) | 主な決裁事項 |
|---|---|---|---|---|
| **0 ガバナンス基盤** | ルールが先に立つ | 営業版不変条項・★SP-C 一式・会議体規程を文書化し、第0回(UC-0)で決裁。**ツール実装ゼロでも成立**(文書+会議だけで開始できる) | ★SP-C 一式 active | AI 利用方針 / 価格権限 / ブランドトーン / 開示範囲 / 委任規程 / ペルソナ編成 |
| **1 単一顧客・単一担当のドラフト支援** | 品質の実証 | 1顧客分の SP-K 決裁 → 人格生成 → **ドラフトのみモード**で運用(UC-1 を送信抜きで回す) | ドラフトのみモードで一定件数連続して「orphan 0・ゲートすり抜け 0・重大インシデント 0」+下書き採用率が基準到達(本体の24時間無人稼働試験の対応物) | 実送信解禁の判断 / 品質基準値 |
| **2 顧客ポリシーの自動起案** | 履歴の資産化 | メール取り込み・通話書き起こし → SP-K draft 自動起案の定常化(UC-2a) | **個人情報統制の決裁**(Q-05)+専門家レビューが前提条件 | データ利用範囲 / マスキング方針 / 外部 LLM 送出方針 |
| **3 多顧客・多担当展開** | 委任構造の実証 | 委任規程の本格運用、担当引き継ぎ(UC-2c)の実証 | 権限超過案件のエスカレーションが設計どおり回ること / 引き継ぎ実例の評価 | 委任範囲の見直し / 担当別の運用差の扱い |
| **4 経営会議の定常運用** | 改善ループ | KPI → 多視点審議 → SP-C/SP-S 改定提案 → 経営トップ決裁(UC-4)。Q-07 を採るなら Phase 1 から並走 | 月次サイクルが2〜3回安定して回ること | 会議サイクル / KPI 定義 |
| **5 プロモーション(一斉系)** | 最高リスクは最後 | セグメント抽出 → 企画審議 → 顧客別パーソナライズ → 配信(UC-3) | **一斉配信の承認方式の決裁**(Q-03)が解禁条件 | 承認方式(全件 / サンプリング+統制) / 配信規模の上限 |

## 4. フェーズゲート基準(たたき台 — 数値は入れない)

ゲートに使う指標の候補。**しきい値は意図的に書かない**(決めるのは会議。
たたき台値のハードコード禁止の文書版):

- 下書き採用率(無修正承認率)と修正率の傾向
- 送信ゲート拒否率とその内訳(拒否理由コード別)
- 根拠連鎖 orphan 件数(これは常に 0 が条件。しきい値の余地なし)
- ポリシー鮮度(review_after 超過の SP-K 件数)
- インシデント件数(誤送信・開示事故・veto すり抜け)
- 担当の体感(承認負荷・速度向上の定性評価 — 形骸化の兆候もここで拾う)

## 5. 最初の一歩 — 本リポジトリ内で今すぐできること

実装ゼロのまま、本リポジトリのシナリオ基盤で構想を前に進められる:

- **document-review シナリオ**で本構想文書一式を3人格レビューにかけ、抜けを潰す
- **deliberation シナリオ**で未決定論点([05](05_リスク・論点.md) §9 の Q)を1件ずつ合議する
- **persona-test シナリオ**で営業ペルソナ編成案(02 §7)の個性が出るか試す
- PoC 着手・別リポジトリ新設の判断は、アイデアカードを `proposed` へ昇格させる時の
  意思決定として扱う(Q-08)

## 6. 本ロードマップの見直し

- アイデアカードの status 変更時(shaping → proposed 等)に必ず見直す
- TradeCouncil 本体の Phase 進行(L2 知能層・news パイプライン・council_runner の実装)で
  流用できる部品が増えた時にも見直す(対応表 02 §2 の「構想」列が具体化する)
