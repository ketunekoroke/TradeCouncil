# リスク・論点

| 項目 | 内容 |
|---|---|
| 位置づけ | 構想(アイデア)。docs/ の一次仕様ではなく、TradeCouncil 本体の仕様・ポリシー・挙動を一切変更しない |
| 効力 | 非規範。本書は論点整理であり、**法的助言ではない**。個人情報・法令(§2)は専門家レビューを前提とする |
| 流用元 | TradeCouncil エグゼクティブ概要「主なリスクと統制の対応」の様式 |

## 1. 評価の枠組み(サマリ)

| 想定リスク | 主な統制 | 残余リスク(統制しきれない部分) | 詳細 |
|---|---|---|---|
| 個人情報・法令違反 | データ最小化(生ログをポリシーへ転記しない)・マスキング層・専門家レビュー | 法解釈の変動、委託先(LLM 事業者)の運用 | §2 |
| 誤送信・無権限の約束 | 送信ゲート・人間承認・ドラフトのみモード・キルスイッチ | 承認者自身の見落とし | §3 |
| ハルシネーション(事実でない実績・仕様) | 根拠メモ(出典必須)・ゲートの事実照合 | 出典自体の誤り | §4 |
| 承認の形骸化(自動化バイアス) | 逆 KPI 監視・ランダム精読サンプリング | 品質向上そのものが形骸化を誘発する構造 | §4 |
| 操作的コミュニケーション | 不変条項⑥・コンプライアンス人格の veto | 「操作」の境界の曖昧さ | §5 |
| 現場の不信(監視ツール化) | 担当自身が SP-K の決裁者である設計 | 評価制度との接続次第で再燃 | §6 |
| データ品質(古い・例外的なノウハウの混入) | 悪パターン退役・review_after 必須 | 過去データの偏り | §7 |

## 2. 個人情報・法令(最優先)

> 本節は論点整理であり法的助言ではない。**専門家(弁護士・個人情報保護の実務家)の
> レビューを、アイデアカードを `proposed` へ昇格させる前提条件とする。**

- **利用目的の整合**: メール・通話録音は個人データを含む。取得時に特定・通知した利用目的と
  「AI による解析・顧客ポリシー生成・文面生成」への利用が整合するか。不足するなら
  目的の変更・通知・同意の要否を整理する
- **外部 LLM への送出の整理**: API 送信が第三者提供か委託か、事業者のデータ利用条項
  (学習利用の有無・保持期間)の確認。機密区分ごとに送出可否を分ける設計(Q-05)
- **通話録音の同意**: 録音・書き起こしの同意取得の運用(相手方・自社従業員の双方)
- **従業員側のプライバシー**: 担当者のメール解析は労務上の配慮も要る(本人への説明と合意。
  §6 の不信問題と直結)
- **データ最小化の設計**(02 §8 と呼応): 生ログは履歴 DB に閉じ、SP-K には抽象化した
  方針のみを書く。**評価情報・要配慮個人情報を SP-K に書かない**ことを設計原則にする
- **開示請求への対応**: 保有個人データの開示・訂正・利用停止請求に対し、顧客ポリシー
  (SP-K)自体が開示対象になり得ることを織り込む(「顧客が読んでも成立する書き方」は
  §5 の倫理面とも整合する)
- **マスキング・仮名化の挿入点**: 履歴取り込み → 解析 → 起案のどこで個人識別子を
  落とすか(02 §9 の新規要素)

## 3. 誤送信・ブランドリスク

| リスク | 統制 | 残余リスク |
|---|---|---|
| 宛先誤り(別顧客への送信) | 送信ゲートの宛先ドメイン照合・担当顧客との突合 | 同一顧客内の宛先人違い |
| 無権限の約束(価格・納期・仕様) | 約束表現の検出+委任範囲チェック(02 §4) | 検出器をすり抜ける婉曲表現 |
| トーン事故(ブランド毀損) | SP-C ブランドトーン+コンプライアンス人格レビュー | 文脈依存の不適切さ |
| **別顧客情報の混入(クロスコンタミ)** | 顧客別エージェントの分離(顧客ごとに参照範囲を物理的に限定)+開示範囲検証 | 共有ノウハウ(SP-S)経由の間接漏えい |

- 顧客別エージェント分離は人格調整のためだけでなく、**参照可能なデータの境界**としても
  機能させる(あおぞら商事のエージェントは、みなと精機(架空)の履歴を読めない)
- 統制が効かない残余リスクは最終的に人間の承認に残る。だからこそ §4 の形骸化対策が
  本構想の急所になる

## 4. ハルシネーションと承認の形骸化

### ハルシネーション

- リスク: 事実でない導入実績・仕様・価格が、もっともらしい文面に混入する
- 統制案: 下書きの**事実主張に出典を要求**する(根拠メモに「この実績は契約 DB #xxx」
  のように participating ソースを紐付ける — 決裁レコード basis_refs の文面版)。
  出典のない事実主張は送信ゲートが警告する

### 承認の形骸化(自動化バイアス)— 独立の重大リスク

下書きの品質が上がるほど、人間は読まずに承認するようになる。**人間決裁を統制の要に
据える本構想にとって、これは構造的な急所**である(本体には無い、営業版固有の最重要リスク)。

対策候補(確定は決裁):

- **逆 KPI 監視**: 修正率・却下率の異常な低下、承認までの時間の異常な短縮を
  「品質向上」ではなく**形骸化の兆候**としてもアラートする
- ランダム精読サンプリング(承認済み送信の事後監査)
- 高リスク文面(初回連絡・価格提示・クレーム対応)だけ承認 UX を重くする(差分強調・
  根拠メモの確認必須化)

## 5. 顧客別人格の倫理 — パーソナライズと操作の境界

- **境界線の問題**: 「顧客に分かりやすく・心地よく伝える」(パーソナライズ)と
  「顧客の意思決定を歪める」(操作)の境界はどこか。不変条項⑥は後者を禁止するが、
  条項だけで境界を完全には定義できない。SP-C のコンプライアンス基準で具体化し続ける
  運用が必要(コンプライアンス人格の veto の主戦場)
- **AI 関与の対外開示**: 3水準 — 出さない / 聞かれたら答える / 明示する。
  どれを選ぶかは経営トップの決裁(Q-02)。なお不変条項⑥により**社内的な隠蔽**
  (担当にすら AI 関与を伏せる等)は選択肢にない
- **「人格モデル化」への心理的抵抗**: 顧客は「自分専用の AI 人格が作られている」ことを
  知ったらどう感じるか。§2 の開示請求対応とあわせ、**SP-K は顧客本人が読んでも
  成立する書き方**(事実と対応方針のみ。人物評・推測を書かない)を設計原則にする

## 6. 組織・人の論点

- **監視ツールと受け取られる懸念**: メール解析・修正差分の記録は、使い方を誤ると
  「営業の一挙一動の監視」になる。緩和の設計: ①担当自身が SP-K の決裁者である
  (自分の顧客知見の主導権は担当にある)②修正差分の一次用途をポリシー改善に限定し、
  個人評価への転用は別途の合意なしに行わない(SP-C で明文化する論点)
- **ノウハウを差し出すインセンティブ**: トップ営業ほど「自分の強みの源泉」を出す動機が弱い。
  評価制度との接続は本構想のスコープ外だが、論点として記録する(引き継ぎ資産化 UC-2c の
  価値を当人にどう還元するか)
- **承認負荷と教育コスト**: 担当には「指示の出し方」「根拠メモの読み方」の習熟が要る。
  導入初期はむしろ遅くなる期間があることを前提に置く(Phase 1 のゲートを焦らない)

## 7. データ収集の実務

- メール連携の接続形態(社内メール基盤との API 連携方式)は**未定**。送信ゲートを
  確実に経由させられる構成(=ゲートを迂回する送信経路を作らない)が選定の必須条件
- 通話書き起こしの精度・コスト・同意運用(§2)
- 既存 CRM / SFA との真実源の競合: 顧客マスタ・案件状態は既存システムが真実源、
  SalesCouncil は**コミュニケーション知見の真実源**、と分担する(置き換えない — 01 §6)
- **過去データの品質**: 例外対応・期限切れの値引き・退職者の癖が「ノウハウ」として
  学習される汚染リスク → 自動起案はあくまで draft、決裁で濾過する+悪パターン退役
  (02 §8)が対抗策
- 退職者・異動者のメールデータの取り扱い(保持期間・利用範囲)

## 8. 技術的論点(構想レベル — 深入りしない)

別リポジトリ設計時の主要課題として登録のみ:

- **マルチユーザー・マルチテナント**: 本体は単一オーナーのローカル CLI 前提。
  営業組織では認証・権限・同時利用が必要になる(最大のアーキテクチャ差分。02 §9)
- 顧客数 × エージェントのコストとスケール(人格は生成ビューなので保管は軽いが、
  推論コストは利用量に比例する)
- 機密区分ごとの LLM 使い分け(オンプレ・リージョン・事業者の選択。Q-05 と連動)

## 10. 撤退基準 — このアイデアをやめる条件

検証で棄却できる形に書いておく(戦略カード「仮説は反証可能に」の適用)。
以下のいずれかが成立したら、parked / retired を正面から検討する:

1. **品質**: ドラフトのみモード(Phase 1)で、下書き採用率が改善傾向に乗らないまま
   推移する(担当が「自分で書いた方が速い」状態が解消しない)
2. **法令**: 専門家レビューの結果、外部 LLM 利用・履歴解析が現実的なコストで
   適法に構成できない(Q-05 が「送出しない」しか取れず、ローカル構成のコストが見合わない)
3. **統制**: 承認の形骸化(§4)が対策を入れても統制できない — 人間決裁が事実上
   機能していないなら、本構想の安全性の前提が崩れる
4. **組織**: 現場の不信(§6)が解消せず、SP-K の決裁・修正フィードバックが回らない
   (ポリシーが鮮度を失い、fail-closed が常態化する)

撤退しても、決裁済みポリシー文書(SP-C のガバナンスコード・ブランドトーン)と
プレイブックは**人間用の資産としてそのまま残る** — ルールを先に作る進め方(Phase 0)は
撤退時の損失を最小化する設計でもある。
