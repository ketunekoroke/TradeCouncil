# SalesCouncil(仮称)— 営業支援ガバナンス・フレームワーク構想

| 項目 | 内容 |
|---|---|
| 位置づけ | 構想(アイデア)。docs/ の一次仕様ではなく、TradeCouncil 本体の仕様・ポリシー・挙動を一切変更しない |
| 効力 | 非規範。本構想内の数値・ポリシー案・人格編成はすべてたたき台であり、決裁されたものは存在しない |
| 流用元 | TradeCouncil のガバナンス設計(`docs/02_基本設計書.md` §1.5、`docs/03_運営規程・第0回アジェンダ.md`)を営業ドメインへ翻訳 |

## 1. 一言で

会社の営業方針・ノウハウ・顧客ごとの対応知見を「会議と決裁を経たポリシー」として蓄積し、
AI エージェントが顧客別に調整された人格でメール文面などを下書きする営業支援システム。
**AI は下書きと提案まで。顧客への送信・約束は人間(営業担当)のみが行う。**

TradeCouncil で実証した統制構造 — **No Policy, No Trade** — の営業ドメイン移植であり、
合言葉は **No Policy, No Send**(必須ポリシーが決裁されるまで1通も下書きしない)。

## 2. 位置づけ

- 本フォルダは構想のみ。コード・設定は一切実装しない
- 実装する場合は**別リポジトリを新設**する想定(本リポジトリは流用元の参照に留める)
- 例示に登場する企業名・人名・数値はすべて架空・たたき台

## 4. 用語と対応の最小セット

| 用語 | 意味(括弧内は TradeCouncil の対応物) |
|---|---|
| No Policy, No Send | 必須ポリシーが active でない顧客・領域では下書き・送信支援を拒否する(No Policy, No Trade) |
| 送信ゲート | 顧客に到達する文面の手前にある唯一の機械検証関門(risk_guard) |
| SP-C / SP-S / SP-K | 会社ポリシー / 営業ノウハウ / 顧客別ポリシーの三層(ポリシーレジストリ P-01〜 の営業版) |
| 顧客別エージェント | 決裁済み SP-K から自動生成される顧客専用人格(`config/generated/` の生成ビューと同じ扱い) |
| ドラフトのみモード | 送信せず下書き品質だけを評価する稼働モード(paper モード) |
| communication_decisions | 「なぜこの文面か」の根拠レコード。全送信が遡及できる(trade_decisions) |

詳細な対応表は [02_TradeCouncil対応設計](02_TradeCouncil対応設計.md) §2。
コアの仕組みそのもの(ドメイン非依存)の解説は [../governance-core/](../governance-core/README.md)。
