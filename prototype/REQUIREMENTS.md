# 要件一覧(Requirements)

MAGI 合議システム プロトタイプ(Phase 0)の要件を一覧化したもの。
**一次資料は [DOCS.md](DOCS.md)**。本ファイルは「何を満たすべきか」を ID 付きで棚卸しする管理表。

- 優先度: **必須**(これが無いと成立しない) / **推奨**(品質・運用上あるべき) / **任意**(あると良い)
- 関連: 実装は [FEATURES.md](FEATURES.md)、検証は [TESTCASES.md](TESTCASES.md) を参照
- 機能を追加・変更したら、本ファイル → FEATURES → TESTCASES を併せて更新する

---

## SC. シナリオ・アーキテクチャ

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-SC01 | 3人格を用途シナリオごとに異なるプロトコルで動かせる(合議 / 資料レビュー …) | 必須 |
| REQ-SC02 | 各シナリオは独立した1ファイル(`scenarios/<name>.md`)で定義する | 必須 |
| REQ-SC03 | ファシリテーター(`CLAUDE.md`)が利用モードで適用シナリオを選ぶ(兆候語の判定表/判別不能時は確認) | 必須 |
| REQ-SC04 | 人格・LLMバックエンド・メディア入力・召喚ルールは全シナリオ共通で、`CLAUDE.md` を唯一の出典とする(シナリオに重複させない) | 必須 |
| REQ-SC05 | シナリオごとに出力先ディレクトリを分ける(合議=`<root>/deliberations/` / 資料レビュー=`<root>/reviews/` / ブレスト=`<root>/brainstorms/` / 人格テスト=`<root>/persona-tests/`。`<root>` は `local/` か `sharepoint/`) | 推奨 |
| REQ-SC06 | どの人格がどの backend/model で動いたかを各シナリオの成果物冒頭に明記する | 推奨 |

## D. 合議シナリオ(deliberation)

`scenarios/deliberation.md`。議題に3人格が議論し合意形成するシナリオの要件。

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-D01 | 価値観で分かれた3人格(MELCHIOR=論理 / BALTHASAR=関係性 / CASPER=直感)が議論する | 必須 |
| REQ-D02 | ファシリテーター(メインセッション)が論題整理・進行・合意形成・成果物生成を担う | 必須 |
| REQ-D03 | 議題の重要度に応じ Lite / Standard / Full の3モードを選べる | 必須 |
| REQ-D04 | Round 0〜9 のプロトコルに沿って進行する(モードごとに含むラウンドが変わる) | 必須 |
| REQ-D05 | 投票は確信度を加味し、反対票・少数意見を最終回答に必ず残す | 必須 |
| REQ-D06 | 各人格は「意図的な弱み」を保持し、予定調和に丸めない | 必須 |
| REQ-D07 | claude 人格間は SendMessage で直接対話できる(使えない実行環境ではファシリテーター仲介=ダイジェスト/`--history` にフォールバック) | 推奨 |

## R. 資料チェック&リバイス シナリオ(document-review)

`scenarios/document-review.md`。資料をレビューし指摘+改訂版を出すシナリオの要件。

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-R01 | 3人格の価値観レンズをレビュー観点に投影する(MELCHIOR=正確性/論理 / BALTHASAR=読者/伝わりやすさ / CASPER=訴求力/声)。専門の編集者・校正者には作り変えない | 必須 |
| REQ-R02 | Round 0 で資料の素性(種類/目的/対象読者/トーン)と改訂の制約(不可変箇所・分量・リライト強度)を確定する | 必須 |
| REQ-R03 | 各人格が同じ資料を独立にレビューし、重要度・箇所・問題・修正提案つきで指摘する | 必須 |
| REQ-R04 | 深度モード Quick / Standard / Deep を選べる(相互レビュー・優先度合意・再検証の有無が変わる) | 必須 |
| REQ-R05 | 衝突する指摘をファシリテーターが裁定し、must/should/nice/見送りに分類する | 必須 |
| REQ-R06 | 合意した指摘を反映した改訂版を**元と同形式**で生成し、変更履歴を記録する | 必須 |
| REQ-R07 | 成果物として指摘レポートと改訂版の両方を出す(指摘だけ・改訂だけにしない) | 必須 |
| REQ-R08 | 未採用の指摘とその理由を成果物に残す(少数意見の保持に相当) | 推奨 |
| REQ-R09 | 原文を尊重し、指摘を捏造せず、合意範囲を超えた全面リライトをしない | 必須 |
| REQ-R10 | docx 等書式付き資料の改訂版は原則「原本コピー編集」で体裁・画像を保ち、全面再構築は構造変更時の最終手段とする(失う体裁を Round 0 で確認) | 推奨 |

## BR. ブレストシナリオ(brainstorm)

`scenarios/brainstorm.md`。テーマにアイデアを発散・評価し上位案を磨くシナリオの要件。

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-BR01 | 3人格の価値観レンズを**アイデアの発散と評価の両方**に投影する(MELCHIOR=実現可能/体系 / BALTHASAR=人中心/持続 / CASPER=大胆/独創)。専門の企画者・コンサルには作り変えない | 必須 |
| REQ-BR02 | Round 0 で目的・制約・成功基準・評価軸(3〜6)と発散ルール(判断後回し/量優先/突飛歓迎/build-on)を確定する | 必須 |
| REQ-BR03 | モード Quick / Standard / Deep を選べる(発散巡数・プレモーテムの有無が変わる)。発散巡数はモード固定で、新案が尽きたら早期収束する(都度指定でも無制限ループでもない) | 必須 |
| REQ-BR04 | 各人格が独立に大量発散し(質より量)、アイデアマップ化(クラスタ化・白地の明示)してから build-on・掛け合わせ・空白埋めで二次発散する | 必須 |
| REQ-BR05 | ショートリストの各アイデアを全人格が評価軸で 0〜10 採点し一言根拠を付す | 必須 |
| REQ-BR06 | スコア集計はレンズ別内訳を保持し(単純平均で潰さない)、上位 N に加えて評価が割れた尖り案を別枠で残す(少数意見の保持に相当) | 必須 |
| REQ-BR07 | 上位案を3レンズ協働でブラッシュアップ(CASPER訴求×MELCHIOR実現×BALTHASAR人/持続。具体化・弱点対処・ネクストステップ)。Deep ではプレモーテムで事前にストレステストする | 必須 |
| REQ-BR08 | 成果物としてアイデア集・アイデアマップ(Mermaid)・評価マトリクス・上位案を出す。アイデア・スコアを捏造しない | 必須 |
| REQ-BR09 | 発散と収束を分離する(発散フェーズでは批判させない)。意図的な弱みを保持し、レンズの差を発散にも評価にも残す | 推奨 |

## PT. 人格テストシナリオ(persona-test)

`scenarios/persona-test.md`。同一依頼への出力差で人格の個性・調整を検査するメタ・シナリオの要件。

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-PT01 | 同一のプローブ(依頼)を全人格に等しく独立に投げ、互いの出力を見せずに答えさせる(議論させない=個性を分離) | 必須 |
| REQ-PT02 | 固定のプローブ・バッテリ(判断 / 好奇心 / 弱み / ドメイン非依存 / 反応)を持ち、回帰テストとして再現可能にする(ユーザーは追加可、既定は不変) | 必須 |
| REQ-PT03 | 各人格の定義(レンズ / 好奇心の向き / 意図的な弱み / 声)をルーブリックに、期待と照合して人格別に判定する。弱みが出るのは仕様どおりと評価する | 必須 |
| REQ-PT04 | 識別性チェック(2人格が似すぎ=個性の潰れ・混ざりを検出して警告する) | 必須 |
| REQ-PT05 | backend は既定で各人格の設定どおり(混在可)、任意で同一 backend に揃えられる。どの人格がどの backend/model で動いたかを成果物に明記する(混在時は出力差にモデル差が混じりうると注記) | 推奨 |
| REQ-PT06 | モード Quick / Standard / Deep を選べる(プローブ範囲・複数 backend・ベースライン回帰の有無が変わる) | 推奨 |
| REQ-PT07 | (任意)過去の実行をベースラインに差分比較し、調整で狙った変化が出たか・他の特性が壊れていないかを確認する | 任意 |
| REQ-PT08 | 成果物として使用した設問(P1〜P5 の文面)・差分マトリクス・人格別判定を出す。出力を期待に寄せて歪めず、中身・判定を捏造しない | 必須 |

## P. 人格定義

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-P01 | 人格は専門分野ではなく価値観・判断軸で定義する(専門ペルソナにしない) | 必須 |
| REQ-P02 | 人格は frontmatter に `name` / `description` / `backend` / `model` を持つ | 必須 |
| REQ-P03 | 人格定義の本文がそのままシステムプロンプトになる(フロントマター・メタコメントは除外) | 必須 |
| REQ-P04 | 3人格は共通の「好奇心・興味」を持ち、向く対象と強度がレンズごとに屈折する(MELCHIOR=仕組み・因果 / BALTHASAR=人・関係 / CASPER=新奇・体験)。好奇心が向きにくい対象は各人格の「意図的な弱み」と呼応させる(弱みは削らない)。対象は粗く定義し、専門ペルソナにしない | 推奨 |

## B. LLM バックエンド選択

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-B01 | 各人格ごとに Claude / ChatGPT(OpenAI)/ Gemini(Google)を frontmatter(`backend`/`model`)で選べる | 必須 |
| REQ-B02 | `backend: claude` は通常召喚、`openai`/`gemini` は各ブリッジ経由で動かす | 必須 |
| REQ-B03 | claude / openai / gemini 人格は混在でき、議論プロトコル(ラウンド/投票/採点)は不変 | 必須 |
| REQ-B04 | どの人格がどの backend/model で動いたかを成果物に明記する(再現性) | 推奨 |
| REQ-B05 | openai/gemini 人格はステートレスなので、必要文脈を毎ラウンド渡して継続性を保つ(stdin/`--input` への織り込み、または `--history` での多ターン会話履歴) | 必須 |
| REQ-B06 | openai/gemini 人格の拒否/空応答時は自動で再試行する(既定1回、`MAGI_GEN_MAX_RETRIES`)。続く場合はファシリテーターが人格定義を変えず指示を言い換えて再実行する | 推奨 |
| REQ-B07 | プロバイダ間で共通する処理(キー解決/抽出/HTTP等)は共有モジュールに集約する | 推奨 |
| REQ-B08 | 過負荷/モデル不在(429/5xx/404・model_not_found)で primary が失敗した場合、指定があれば代替モデルへ1回フォールバックできる(人格定義は変えず、実際に使ったモデルを成果物に明記) | 推奨 |

## F. ファイル入力

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-F01 | 画像(jpg/png/gif/webp)を全 backend でネイティブに扱える | 必須 |
| REQ-F02 | PDF を openai/gemini 人格でネイティブに扱える(テキストを持つ PDF) | 必須 |
| REQ-F03 | Office(docx/xlsx/pptx)を準ネイティブに扱える(ローカルでテキスト抽出。docx は段落と表をドキュメント順に保つ) | 推奨 |
| REQ-F04 | テキストファイル(txt/md/csv/json 等)を本文として注入できる | 推奨 |
| REQ-F05 | 同一ファイルを全人格に等しく渡せる(claude/openai/gemini で挙動が対称) | 必須 |
| REQ-F06 | 同じ画像/PDF を多ラウンドで使う場合、アップロードして file_id を使い回せる | 推奨 |

## SP. SharePoint 連携(任意)

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-SP01 | 入出力ファイル(入力資料・成果物)を SharePoint で共有できる。`scripts/sharepoint.py`(Graph API)が遠隔とローカルミラーを同期する | 任意 |
| REQ-SP02 | オン/オフで アクティブ root が `local/`(オフ)↔ `sharepoint/`(オン)に切り替わる。オフ時は通信せず従来挙動。設定は `settings.local.json` の env(`MAGI_SHAREPOINT_ENABLED` 等)→ `sharepoint.config.json` の順で解決(env が Git 追跡外でオンオフ・接続先を保持できる) | 任意 |
| REQ-SP03 | 両 root は `input / media-output / reviews / deliberations / brainstorms / persona-tests` の構成が同一。アクティブ root は `sharepoint.py root` で解決する | 任意 |
| REQ-SP04 | 認証はアプリ(クライアントシークレット)。`MAGI_SHAREPOINT_TENANT_ID`/`..._CLIENT_ID`/`..._CLIENT_SECRET` を他ブリッジと同じ作法(env → `settings.local.json`)で解決し、コミットしない | 任意 |
| REQ-SP05 | `pull`(遠隔→ローカル)/`push`(ローカル→遠隔)/`info`(SharePoint URL)を提供し、成果物提示時にローカルパスと URL の両方を示す | 任意 |
| REQ-SP06 | SharePoint 連携も標準ライブラリだけで動く(HTTP は共有モジュール経由、新規 pip 依存なし) | 推奨 |
| REQ-SP07 | Azure アプリ登録のセットアップ手順を `documents/` にマニュアル化し、README/DOCS から参照する。記載内容(env 名・必要権限・コマンド)はコード/設定と一致させる | 推奨 |
| REQ-SP08 | セットアップ手順は外部仕様(Entra ID 画面・Microsoft Graph)に依存するため、定期的に最新の公式情報を確認し、変更があればマニュアルを更新する | 任意 |

## N. 非機能

| ID | 要件 | 優先度 |
|---|---|---|
| REQ-N01 | ブリッジは標準ライブラリで動く(Office 抽出時のみ追加ライブラリ) | 推奨 |
| REQ-N02 | クロス OS(Windows/macOS/Linux)で動作し、日本語を UTF-8 で**入出力**する(stdin 入力・stdout 出力とも) | 必須 |
| REQ-N03 | エラーはトレースバックでなく整形メッセージで返す | 推奨 |
| REQ-N04 | トークン消費が通常の数倍になる点をドキュメントで明示する | 任意 |
| REQ-N05 | 一過性の HTTP エラー(429 / 5xx / 接続タイムアウト)は指数バックオフ + ジッタで自動リトライする(`Retry-After` 尊重、回数・タイムアウトは `MAGI_HTTP_*` で調整可) | 推奨 |
