# シナリオ(scenarios/)

価値観で分かれた3人格(MELCHIOR / BALTHASAR / CASPER)を、**用途に応じてどう動かすか**を
定義したプロトコル集。各シナリオは独立した1ファイルで、ファシリテーター(`CLAUDE.md`)が
ユーザーの最初の発言から適用するシナリオを選び、その内容に従って進行する。

> 役割・人格・LLMバックエンド選択・メディア入力・召喚ルール・ファシリテーターの心得など
> **全シナリオ共通の作法は `../CLAUDE.md`** にある。各シナリオファイルは「そのシナリオ固有の
> ラウンド構成・モード・成果物」だけを定義する(共通部分を重複させない)。

## シナリオ一覧

| シナリオ | ファイル | 何をするか | 出力先 |
|---|---|---|---|
| **合議**(deliberation) | [deliberation.md](deliberation.md) | 議題に対し3人格が議論し、確信度を加味して合意形成する | `../<root>/deliberations/` |
| **資料チェック&リバイス**(document-review) | [document-review.md](document-review.md) | 持ち込まれた資料を3レンズでレビューし、指摘レポートと改訂版を出す | `../<root>/reviews/` |
| **ブレスト**(brainstorm) | [brainstorm.md](brainstorm.md) | テーマに3レンズでアイデアを発散・評価し、マップと上位案を出す | `../<root>/brainstorms/` |
| **人格テスト**(persona-test) | [persona-test.md](persona-test.md) | 同一依頼への出力差で人格の個性・調整を検査する(QA/回帰) | `../<root>/persona-tests/` |

> `<root>` は `local/`(既定)または `sharepoint/`(SharePoint 連携時)。→ `CLAUDE.md`「SharePoint 連携」

## ファシリテーターによるシナリオ選択

利用モード(開発作業ではない)と判定したら、`CLAUDE.md`「シナリオの選択」の判定表に従って
シナリオを1つ選び、ユーザーに「このシナリオで進めます」と告げてから該当ファイルを開始する。
判別がつかない場合はユーザーに確認する。

| シナリオ | 選択の兆候となる言葉 |
|---|---|
| 合議 | 「議題」「相談」「どうすべき」「合議」「賛否」「迷っている」 |
| 資料チェック&リバイス | 「レビュー」「添削」「校正」「チェックして」「リバイス」「この資料を見て」「直して」「改訂」 |
| ブレスト | 「ブレスト」「ブレインストーミング」「アイデア出し」「ネタ出し」「発散」「企画」「アイデアを広げ」 |
| 人格テスト | 「人格テスト」「人格チェック」「人格を比較」「同じ依頼で人格の違い」「(人格)調整の確認」「個性が出ているか」 |

## 新しいシナリオを追加するには

1. このディレクトリに `scenarios/<name>.md` を新規作成(既存ファイルの構成を踏襲)
   - そのシナリオ固有の「モード / ラウンド / 成果物 / 固有の心得」を書く
   - 共通部分は重複させず `../CLAUDE.md` を参照する
2. 本 README の「シナリオ一覧」と「選択の兆候」表に追記
3. `../CLAUDE.md`「シナリオの選択」の判定表と「出力ディレクトリ」表に追記
4. 出力先ディレクトリを新設する場合は `../.gitignore` に除外ルールを追加
5. `../DOCS.md`「シナリオ」章、`../REQUIREMENTS.md` / `../FEATURES.md` / `../TESTCASES.md` を同期
6. 詳しい手順は `../DEVELOPMENT.md`「パターン6: シナリオを追加する」を参照
