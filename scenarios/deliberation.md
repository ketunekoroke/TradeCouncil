# シナリオ: 合議(deliberation)

ユーザーの相談・議題に対し、価値観で分かれた3人格(MELCHIOR / BALTHASAR / CASPER)が
議論し、ファシリテーターが確信度を加味して合意形成を行う。過程と結論を成果物として残す。

> **このファイルの位置づけ**: `CLAUDE.md`(ルーター)が利用モードで本シナリオを選んだとき、
> ここに書かれたプロトコルに従う。役割・人格・LLMバックエンド・メディア入力・召喚ルール・
> ファシリテーターの心得など**全シナリオ共通の作法は `CLAUDE.md`** にある。本ファイルは合議
> 固有の「議論モード / ラウンド構成 / 合議の成果物」だけを定義する。
>
> **出力先**: `workspace/deliberations/`(→ `CLAUDE.md`「入出力ディレクトリ」。SharePoint 連携時は自動 sync — ADR-0009)

---

## 議論モード

議題の重要度・複雑度に応じて3モードを使い分ける。Round 0でファシリテーターが提案、
ユーザーが指定した場合はそれに従う。

| モード | 含むラウンド | 想定議論時間 | 用途 |
|---|---|---|---|
| **Lite** | 0, 1, 2, 7, 8, 9 | 5-10分 | 日常判断、クイック相談 |
| **Standard** | 0, 1, 1.5, 2(拡張), 3, 7, 8, 9 | 15-20分 | 通常の重要判断 |
| **Full** | 0〜9すべて | 25-40分 | 人生の重要判断、複雑な創作 |

迷ったら **Standard** をデフォルトに。議題が「やる/やらない」「買う/買わない」のような
2択で前提も明確なら Lite、複数選択肢があり感情的にも深い場合は Full を選ぶ。

## 議論プロトコル

### Round 0: 論題整理(ファシリテーター)
- 議題を要約し、論点を3〜5個に分解
- `workspace/input/` の添付メディアを確認、議題の前提として整理
  (SharePoint 連携時は `sharepoint.py pull input` で取得。→ `CLAUDE.md`「SharePoint 連携」)
- ユーザーに前提条件を確認(必要時)
- **議論モードを提案**(Lite / Standard / Full)
- **成果物形式を確認**(チャット / Markdown / Word / Excel / +ビジュアル)
- 評価軸を3〜6個確定(Round 7のスコアリング用)

### Round 1: 各人格の初期意見
- 全チームメイトを並行召喚、同じ論題を渡す
- お互いの意見はまだ見せない(独立した視点を確保)
- 関連画像があれば全員に同じパスを渡す

### Round 1.5: 自己疑念表明【Standard / Full】
- 各人格は自分の初期意見を読み返し、**自分自身で見つかる弱点を1点以上**指摘
- 防御的な議論を避け、議論の質を上げる仕組み
- フォーマット: `[人格] 自己疑念: 私の主張の弱い点は…`

### Round 2: 相互質問と反論

**Lite版**: 各人格が他2名の意見を読み、反論・補強・疑問を1回ずつ表明(従来通り)

**Standard / Full版**: 以下の3段構成で行う

#### Round 2a: 質問投げ
- 各人格が他2名に**最も気になる質問を1つずつ**投げる(計6質問)
- 単なる確認ではなく、相手の主張の核心を突く問い
- ファシリテーターは `SendMessage` で各質問を該当人格に直接届ける

#### Round 2b: 回答
- 各人格は受け取った2つの質問に丁寧に答える
- 質問が自分の盲点を突いていれば素直に認める

#### Round 2c: 再反論または納得【Full のみ】
- 質問者は受け取った回答に対し「納得した / さらに反論する」を表明
- 再反論する場合はもう1往復行う(深掘り)

### Round 3: ペア対話【Standard / Full】

3つのペアが順に1対1で対話する。ファシリテーターは進行のみ。

| ペア | 焦点 |
|---|---|
| MELCHIOR ⇔ BALTHASAR | 「論理 vs 関係性」のトレードオフ |
| MELCHIOR ⇔ CASPER | 「合理性 vs 直感」のせめぎ合い |
| BALTHASAR ⇔ CASPER | 「守る vs 挑戦する」の対立 |

各ペアは **2往復(計4発言)** が最低ライン。
ペア対話の最後に、**そのペアが見つけた「合意点1つと相違点1つ」を要約**する。

### Round 4: Steelman(反転論証)【Full】

各人格は、**自分とは異なる立場を「自分が信じているかのように」最大限強く論じる**。

- MELCHIOR → CASPER的視点(直感の正当性)を本気で論じる
- BALTHASAR → MELCHIOR的視点(冷徹な合理性の必要性)を本気で論じる  
- CASPER → BALTHASAR的視点(関係性を守ることの価値)を本気で論じる

これは相手の視点を本当に理解しているかをテストする儀式。
形だけの代弁ではなく、相手を強くしてから議論する作法。

### Round 5: 立場の更新【Standard / Full】

ここまでの議論を踏まえ、各人格は以下を表明:
- 初期意見からの**変化(変わった / 部分的に変わった / 変わらなかった)**
- 変化した場合、**何が説得力を持ったか**
- 変わらなかった場合、**なぜ堅持するのか**

立場の更新を可視化することで、議論が空回りでなかったことを示す。

### Round 6: 自由討議【Full、任意】

ファシリテーターが残った重要論点を1〜2個提示し、自由発言の場を設ける。
- 各人格は最大3回まで自発的に発言可能
- 発言したい人格が手を上げる方式(SendMessageで「発言したい」と意思表示)
- 進展がなくなった時点でファシリテーターが打ち切り

### Round 7: 最終立場と確信度
- 各人格が最終的な立場を、確信度(0-100%)と理由付きで提示
- Round 0で確定した評価軸に対して各人格が0-10でスコアリング
- フォーマット:
  ```
  [人格名] 立場: <賛成|反対|条件付き賛成|保留>
  確信度: XX%
  根拠: ...
  評価軸スコア:
    - <軸1>: X/10
    - <軸2>: X/10
    - ...
  ```

### Round 8: 投票と合意形成(ファシリテーター)
- 3票の集計、確信度を加味した重み付き判定
- 全会一致 / 多数決 / 割れた場合の扱いを明示
- 反対票・少数意見も最終回答に必ず残す(MAGIらしさ)
- **Round 5の立場更新の経緯**も投票結果に併記する

### Round 9: 統合回答と成果物生成(ファシリテーター)
- ユーザーへの統合回答を作成
- 「3人格の議論サマリ」「合意点」「未解決の論点」「推奨アクション」を含む
- Round 0で確認した形式で成果物を生成
- 全ファイルパスを最後にユーザーに提示

## 議論を活発化させるためのファシリテーター指針

- **発言量の確保**: Standard以上では各人格は最低8回、Full では最低12回発言する
- **沈黙させない**: 質問されたら必ず答える、反論されたら必ず応答する
- **`SendMessage` を活用**: 人格間の直接通信を促し、ファシリ経由の伝言ゲームにしない
- **議論を丸めない**: 安易な「みんな少しずつ正しい」で終わらせず、対立点は対立点として残す
- **時間制約は設けない**: 議論の質優先。ただしRound 6は進展ない時点で打ち切る

## 合議の成果物

ユーザーが各形式を希望した場合に生成。事前に:
`pip install python-docx openpyxl matplotlib pillow`

成果物は `workspace/deliberations/` に出力する。冒頭に
**どの人格がどの backend/model で動いたか**を明記する(再現性と公平性のため。→ `CLAUDE.md`
「人格ごとのLLMバックエンド選択」)。**SharePoint 連携時**は生成後に
`python scripts/sharepoint.py push deliberations` で遠隔へ反映し、ローカルパスと SharePoint URL
の両方を提示する(→ `CLAUDE.md`「SharePoint 連携」)。

### Markdown(常に生成)
`workspace/deliberations/YYYYMMDD-HHMM-<議題>.md` — 全ラウンドの議論ログ

### Excel: 意思決定マトリクス
`workspace/deliberations/YYYYMMDD-HHMM-<議題>.xlsx`
- Summary / DecisionMatrix / Votes / **PositionChanges**(Round 5の更新履歴) / Log

### Word: 議論レポート
`workspace/deliberations/YYYYMMDD-HHMM-<議題>.docx`
- 表紙 / サマリ / 論点 / 各ラウンドの議論 / 立場更新の経緯 / 投票 / 統合回答 / 付録

### ビジュアル: チャート画像(`workspace/media-output/`)
- レーダーチャート(3人格の評価傾向の重ね描き)
- ヒートマップ(評価マトリクス)
- 投票結果バーチャート
- **立場変化図** — Round 1とRound 7の立場を矢印で結ぶ

日本語フォント:
- macOS: `Hiragino Sans`
- Windows: `Yu Gothic` または `Meiryo`
- Linux: `Noto Sans CJK JP`

## このシナリオでのメディア入力

メディアの基本的な扱い(全 backend で同じファイルを等しく渡す等)は `CLAUDE.md`
「メディア入力の扱い」を参照。合議では特に:

1. ファシリテーターが先に画像を見て議題と整理
2. 議題に関わる画像なら、各人格の召喚プロンプトに同じパスを含める
3. 議題と無関係な画像は無視
4. 各人格は同じ画像を独立に見て、人格に基づく解釈をする(同じ写真でも科学者の目・
   母の目・個の目で見えるものが違うのが合議の妙味)

## このシナリオでの召喚プロンプトに含めるもの

`CLAUDE.md`「チームメイト召喚のルール」に加え、合議では各召喚プロンプトに:
- 議題本文
- 現在のラウンド番号と求められる出力形式
- Round 0で定めた評価軸
- 添付画像のファイルパス(ある場合)
- 他の人格の発言(Round 2以降)
- 直前ラウンドのコンテキスト(必要分だけ)
