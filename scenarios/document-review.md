# シナリオ: 資料チェック & リバイス(document-review)

ユーザーが持ち込んだ資料(文章・スライド・レポート等)を、価値観で分かれた3人格
(MELCHIOR / BALTHASAR / CASPER)が**それぞれのレンズで独立にレビュー**し、ファシリテーターが
指摘を統合・裁定したうえで、**指摘レポートと改訂版ドキュメントの両方**を成果物として残す。

> **このファイルの位置づけ**: `CLAUDE.md`(ルーター)が利用モードで本シナリオを選んだとき、
> ここに書かれたプロトコルに従う。役割・人格・LLMバックエンド・メディア/ファイル入力・召喚
> ルール・ファシリテーターの心得など**全シナリオ共通の作法は `CLAUDE.md`** にある。本ファイル
> は資料レビュー固有の「レンズ / レビュー深度 / ラウンド構成 / 成果物」だけを定義する。
>
> **出力先**: `workspace/reviews/`(→ `CLAUDE.md`「入出力ディレクトリ」。SharePoint 連携時は自動 sync — ADR-0009)

---

## このシナリオでの3人格(レビューのレンズ)

人格定義(`.claude/agents/*.md`)は合議と同一。価値観のレンズを**資料レビューに投影**して使う。
人格を専門家(編集者・校正者)に作り変えるのではなく、価値観のレンズをそのままレビュー観点に
向ける(本プロジェクトの「価値観で分ける」原則を維持)。

| 人格 | 価値観レンズ | レビューで見るもの(観点) |
|---|---|---|
| **MELCHIOR**(科学者) | 論理・分析・客観性 | 事実の正確性、データ・出典の妥当性、論理の飛躍・矛盾、定義の曖昧さ、構成の一貫性、主張と根拠の対応、反証可能性 |
| **BALTHASAR**(母) | 共感・関係性 | 対象読者に伝わるか、前提知識への配慮、専門用語・略語の扱い、トーンと敬意、誤解・不安を生む表現、読み手の認知負荷、置き去りにされる読者 |
| **CASPER**(個人) | 直感・欲求・自己実現 | つかみ・冒頭の引き、本当に言いたいことが立っているか、冗長さ・退屈さ、独自性、印象に残るか、書き手の声が生きているか |

各人格の「意図的な弱み」もレビューに作用する(MELCHIORは表現の温度を見落とす、BALTHASARは
必要な厳しい指摘を躊躇しがち、CASPERは正確さより面白さを優先しがち)。だからこそ3レンズを
重ねる。1つのレンズの過剰・欠落を他の2つが補正する。

各レンズの**好奇心**が「どこに引っかかるか」を決める(MELCHIORは仕組み・正確さ、BALTHASARは読者・人、
CASPERは面白さ・新奇さに目が向く。→ `CLAUDE.md`「3つの人格」)。だから3人で読むと拾う指摘の範囲が広がる。

## レビュー深度モード

資料の重要度・分量に応じて選ぶ。Round 0でファシリテーターが提案、ユーザー指定があれば従う。

| モード | 含むラウンド | 用途 |
|---|---|---|
| **Quick** | 0, 1, 4, 6 | 短い資料・軽い校正。3レンズで一巡指摘 → 改訂 → 提示 |
| **Standard** | 0, 1, 2, 3, 4, 6 | 通常の資料。相互レビューと優先度合意を経て改訂 |
| **Deep** | 0, 1, 1.5, 2, 3, 4, 5, 6 | 公開前・重要資料。改訂版の再レビューまで行う |

迷ったら **Standard**。1ページ程度のメール・告知なら Quick、外部公開・契約・重要提案なら Deep。

## レビュープロトコル

### Round 0: 資料受領と方針整理(ファシリテーター)
- **(SharePoint 連携時)** `python scripts/sharepoint.py pull input` で対象資料を取得してから読む
  (→ `CLAUDE.md`「SharePoint 連携」)。無効時は不要。
- 対象資料を読む(`workspace/input/` のファイル、またはパス指定。md/txt/docx/pdf/xlsx/pptx 等)
- **資料の素性を確認**(不明ならユーザーに質問):
  - 種類(レポート / 提案書 / メール / 記事 / スライド / 仕様書 …)
  - 目的(何を達成したい資料か)
  - 対象読者(誰に届けるか。前提知識のレベル)
  - 希望トーン(硬め / 柔らかめ / 中立 …)
- **レビュー観点(評価軸)を確定**。既定は上記3レンズ。ユーザーが軸を追加・限定可
  (例:「正確性だけ見て」「トーンは触らないで」)
- **改訂の制約を確認**:
  - 文字数・分量の制限
  - **変えてはいけない箇所**(引用、固有名詞、決定済みの数値、法的文言 等)
  - トーン・文体を維持するか、変えてよいか
  - リライトの強さ(最小限の修正 / 積極的な書き換え)
- **レビュー深度モードを提案**(Quick / Standard / Deep)
- **成果物形式を確認**(レビューレポート形式 + 改訂版の形式。改訂版は既定で**元と同形式**)

### Round 1: 独立レビュー(3人格並行)
- 全人格を並行召喚し、**同じ資料**(ファイルパス/抽出テキスト)を等しく渡す
- 各人格は他者の指摘を見ず、自分のレンズで独立に指摘を列挙
- 指摘フォーマット(1件ごと):
  ```
  [人格] 指摘#N
  重要度: 高 | 中 | 低
  箇所: <セクション名 / 該当文の引用>
  問題: <何が・なぜ問題か(自分のレンズの観点で)>
  修正提案: <どう直すか。可能なら具体的な書き換え案を「変更前→変更後」で>
  ```
- **資料に実在する箇所だけを指摘する**(存在しない記述を捏造しない)
- 良い点も1〜2個挙げてよい(改訂で壊さないため)

### Round 1.5: 自己点検【Deep】
- 各人格は自分の指摘を読み返し、**過剰指摘・好みの押し付け**がないか自己点検
- 「これは資料の欠陥か、それとも自分の好みか」を1件以上について明示
- フォーマット: `[人格] 自己点検: 指摘#N は好みに寄っている / 重要度を下げる`

### Round 2: 相互レビューと指摘の突き合わせ【Standard / Deep】
- 各人格が他2名の指摘を読み、**補強 / 反論 / 重複の統合**を表明
- **特に衝突する指摘を顕在化させる**。例:
  - MELCHIOR「厳密に条件を書き足せ」 ⇔ CASPER「冗長、ここは削ってテンポを出せ」
  - BALTHASAR「この強い表現は読者を不安にする」 ⇔ MELCHIOR「事実として正確だ」
- claude 人格間は `SendMessage` で直接やり取りする(伝言ゲームにしない)。**SendMessage が使えない
  実行環境では**、ファシリテーターが各人格へ他者の指摘をダイジェスト/`--history` で渡して仲介する
  (openai/gemini 人格は元よりこの仲介方式。→ `CLAUDE.md`「コンテキスト継続」)

### Round 3: 優先度付けと改訂方針の合意【Standard / Deep】
- ファシリテーターが全指摘を統合し**重複を排除**、一覧化(指摘IDを振る)
- **衝突した指摘を裁定**:ファシリテーターが論点を提示 → 該当人格が短く主張 → 合意
- 各指摘を分類:
  - **must-fix**(誤り・重大な欠陥。必ず直す)
  - **should-fix**(直すべき。合意できれば反映)
  - **nice-to-have**(あれば良い。改訂方針と矛盾しなければ反映)
  - **見送り**(好み・範囲外・Round 0の制約に抵触。理由を残す)
- **改訂方針**を1〜2文で確定(全体の方向性。例「正確性を担保しつつ冒頭を簡潔にし、トーンは維持」)

### Round 4: 改訂版の生成(ファシリテーター)
- 合意した指摘(must / should、合意あれば nice)を反映した**改訂版**を生成
- **元と同形式**で出力(Markdown → md、Word → docx、テキスト → txt 等)
- **docx など書式付き資料の改訂版は2方式から選ぶ**(同梱ヘルパー):
  - **原本コピー編集(推奨)**: 変更箇所だけ差し替え、**元の体裁・画像・チャートを保つ**。合意した変更を
    `[{"find":..,"replace":..}]` の JSON にし `python scripts/docx_replace.py 原本.docx -r changes.json -o 改訂版.docx`(原本は不変)。画像・複雑な書式を持つ資料はこちら。
  - **全面再構成(構造を大きく変えるとき)**: 改訂本文を Markdown で作り
    `python scripts/md_to_docx.py 改訂版.md -o 改訂版.docx` で書き出す。論文化など構成自体を組み替える場合に。**元の画像・チャート・複雑な書式は失われる**点に注意し、重要な図版があれば原本コピー編集を選ぶ。
- **原文の意図・書き手の声・トーンを尊重**。Round 0で決めた「変えてはいけない箇所」を厳守
- 合意範囲を超えた**勝手な全面リライトをしない**(リライト強度は Round 0 の指定に従う)
- **変更を記録**(変更履歴):箇所 / 変更前 → 変更後 / 対応する指摘ID / 理由

### Round 5: 改訂版の検証【Deep】
- 改訂版を3人格に渡し再レビュー。各人格は:
  - 自分の指摘が**反映されたか**
  - 改訂で**新たな問題**(意味の変質、トーン崩れ、新たな矛盾)が生じていないか
  - 全体として改善されたか
- 重大な見落とし・改悪があれば Round 4 に**1回だけ差し戻す**

### Round 6: 成果物提示(ファシリテーター)
- レビューレポート + 改訂版 + 変更サマリを生成し、全ファイルパスを提示
- **(SharePoint 連携時)** `python scripts/sharepoint.py push reviews` で成果物を遠隔へ反映し、
  ローカルパスと SharePoint URL(`sharepoint.py info <パス>`)の両方を提示する(→ `CLAUDE.md`)。
- **採用しなかった指摘とその理由も必ず残す**(合議の「少数意見の保持」に相当)

## 召喚のヒント:レンズの当て方とステートレス文脈

全シナリオ共通の召喚ルールは `CLAUDE.md`「チームメイト召喚のルール」。本シナリオ固有の補足。

### レンズの当て方
- **人格定義(`.claude/agents/*.md`)は変えない**。レビューへ向けるのは召喚プロンプト側の役目。
  各人格には、上表「このシナリオでの3人格」の観点を**資料に投影するよう**指示する(校正者・編集者に
  作り変えない)。
- Round 1 の召喚プロンプトに最低限含めるもの:資料(`--file` / パス)、最終ゴール(対象読者・媒体・
  リライト強度)、上表のレンズ別観点、出力フォーマット(`[人格] 指摘#N / 重要度 / 箇所 / 問題 /
  修正提案`)。
- 各人格の「意図的な弱み」を一言添えると効く(例:BALTHASAR には「重要資料では遠慮しすぎない」)。

### ステートレス人格への文脈の渡し方(ラウンド別)
openai / gemini 人格は呼び出しごとに記憶がリセットされる(→ `CLAUDE.md`「コンテキスト継続」)。
各ラウンドで最低限渡すべき文脈:

| ラウンド | その人格に毎回渡すもの |
|---|---|
| Round 1 | 資料(`--file`)、最終ゴール、レンズ別観点、出力フォーマット |
| Round 1.5 / 2 | 上記 + 自分の Round 1 指摘 + 他人格の指摘ダイジェスト(衝突点を明示) |
| Round 3 | (ファシリテーター主導)裁定する論点 + 当該人格の主張 |
| Round 5 | 改訂版 + 自分の元指摘(反映確認のため) |

過去ラウンドは `--history`(その人格の過去発言は `assistant` ロール)で会話履歴として渡せる。
同じ資料を多ラウンド使うなら `upload` → `--file-id` の使い回しでトークンを節約。

## ファシリテーターの心得(このシナリオ固有)

- **原文を尊重する**。書き手の声を消した「無難で平板な文章」に均さない
- **指摘を捏造しない**。資料に実在する箇所についてのみ指摘・修正する
- **改訂は指摘の反映であって作文ではない**。Round 0 / Round 3 で合意した範囲を超えない
- **衝突を丸めない**。「両方少しずつ」で逃げず、裁定して理由を残す
- **全人格に同じ資料を等しく渡す**(レンズの差だけが指摘の差になるように)
- 各人格に「専門家のふり」をさせない。価値観レンズのまま資料に向かわせる

## 成果物

成果物は `workspace/reviews/` に出力する。冒頭に
**どの人格がどの backend/model で動いたか**を明記する
(再現性。→ `CLAUDE.md`「人格ごとのLLMバックエンド選択」)。Word/Excel 生成には
`pip install python-docx openpyxl` が必要。

### レビューレポート(Markdown、常に生成)
`workspace/reviews/YYYYMMDD-HHMM-<資料名>-レビュー.md`
- 資料概要(種類 / 目的 / 対象読者 / トーン / 改訂の制約)
- 使用人格と backend/model、レビュー観点(評価軸)
- **指摘一覧**(観点別 × 重要度別、指摘IDと採否ステータス must/should/nice/見送り 付き)
- **衝突した指摘とその裁定**
- **変更サマリ**(指摘ID / 箇所 / 変更前 → 変更後 / 理由)
- **未採用の指摘と理由**(少数意見の保持)

### 改訂版ドキュメント(元と同形式、常に生成)
`workspace/reviews/YYYYMMDD-HHMM-<資料名>-改訂版.<元の拡張子>`
- 合意した指摘を反映した本文。元が docx なら docx(python-docx)、md なら md
- 必要なら冒頭または末尾に「改訂サマリ」を付す(本文を汚さない形で)

### 任意: 指摘トラッカー(Excel)
`workspace/reviews/YYYYMMDD-HHMM-<資料名>-指摘一覧.xlsx`
- 列: 指摘ID / 人格 / 観点 / 重要度 / 箇所 / 問題 / 修正提案 / 採否 / 理由
- 条件付き書式で重要度・採否を色分けすると見やすい

## このシナリオでのファイル入力

ファイルの基本的な扱い(全 backend で同じファイルを等しく渡す、Office はローカル抽出、
多ラウンドは `upload`→`file-id` 使い回し等)は `CLAUDE.md`「メディア入力の扱い」を参照。
資料レビューでは:

- 対象資料そのものが入力の主役。`workspace/input/` に置くか、パスを直接指定する
  (SharePoint 連携時は `sharepoint.py pull input` で取得。→ `CLAUDE.md`「SharePoint 連携」)
- openai / gemini 人格には `--file <資料パス>` で渡す(docx/pdf/xlsx/pptx も抽出して渡る)
- 多ラウンド(Round 1 → 2 → 5 で同じ資料を再参照)では、一度 `upload` して `file-id` を
  使い回すとトークンを節約できる
- claude 人格には召喚プロンプトに資料パスを含めてネイティブに読ませる
- 全人格に等しく渡す「正本テキスト」が要るときは `python scripts/extract_office.py 資料.docx -o canon.md` で
  本文順 Markdown 化できる(見出し・GFM 表・画像マーカーを保持。docx/pptx/xlsx 対応)
