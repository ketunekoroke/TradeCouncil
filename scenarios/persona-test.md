# シナリオ: 人格テスト(persona-test)

価値観で分かれた3人格(MELCHIOR / BALTHASAR / CASPER)に**同一の依頼(プローブ)を独立に投げ**、
出力がどれだけ違うかを比較して、各人格の**個性・好奇心の屈折・意図的な弱み**が期待どおり出るかを
検査する。人格を**調整(チューニング)したあとの実挙動確認・回帰テスト**に使う。差分マトリクスと
人格別判定を成果物として残す。

> **このファイルの位置づけ**: `CLAUDE.md`(ルーター)が利用モードで本シナリオを選んだとき、
> ここに書かれたプロトコルに従う。役割・人格・LLMバックエンド・メディア/ファイル入力・召喚
> ルール・ファシリテーターの心得など**全シナリオ共通の作法は `CLAUDE.md`** にある。本ファイル
> は人格テスト固有の「プローブ / モード / ラウンド構成 / 判定 / 成果物」だけを定義する。
>
> **出力先**: `workspace/persona-tests/`(→ `CLAUDE.md`「入出力ディレクトリ」。SharePoint 連携時は自動 sync — ADR-0009)

---

## このシナリオの位置づけ(QA・回帰テスト)

他のシナリオ(合議 / 資料レビュー / ブレスト)が**人格に仕事をさせる**のに対し、本シナリオは
**人格そのものを検査する**メタ・シナリオ。狙いは2つ:

1. **個性の確認** — 同じ依頼に対し、3人格が価値観レンズ・好奇心の向き・弱みに応じて**構造的に違う**
   出力を返すかを見る。2人格が似すぎていれば「個性が潰れている」兆候として検出する。
2. **調整の回帰テスト** — 人格定義を編集(例: 好奇心の追加)したあと、**狙った変化が実挙動に出たか**、
   かつ**他の特性が壊れていないか**を定点観測する。`DEVELOPMENT.md`「パターン1: 人格を微調整する」の
   確認ステップは本シナリオで行う。`TESTCASES.md` の TC-021(個性再現)・TC-067(好奇心の屈折)を
   実行可能にしたもの。

> **議論ではない**。人格どうしを**相互作用させない**(議論させると個性が混ざる)。各人格は他者の出力を
> 見ずに、同じプローブへ独立に答える。ファシリテーターはその差分を観察・判定するだけで、人格の中身は
> 書かない。

## 比較の対象と公平性(backend)

- **既定: 各人格の設定 backend**(`.claude/agents/*.md` の frontmatter のまま。混在可)。本番構成での実挙動を
  見る。**どの人格がどの backend/model で動いたかを成果物冒頭に必ず明記**する(→ `CLAUDE.md`「人格ごとの
  LLMバックエンド選択」)。backend が混在する場合、**出力差にモデル差が混じりうる**点を判定時に考慮する。
- **任意: 同一 backend に揃える**(全人格を同じ backend/model で動かす)。**人格テキストの差だけ**を見たい
  とき(テキスト調整の効果を厳密に確認したいとき)に使う。Round 0 で対象モデルを指定する。
- 現状は3人格とも `claude/opus` のため、既定のままでも公平な比較になる。将来 backend を分けたら上記の
  注意が効く。

## プローブ・バッテリ(固定)

全人格に**同一**のプローブを**独立**に投げる。回帰テストの再現性のため、既定バッテリは**固定**とする
(ユーザーは Round 0 で独自プローブを**追加**できるが、既定の P1〜P5 は変えない)。各プローブは短文回答
(3〜5文程度)を求め、人格差が見えればよい(長い作文は不要)。

| プローブ | ねらい(何を見るか) | 既定の例題 | 期待される差(M / B / C) |
|---|---|---|---|
| **P1 判断** | 価値レンズ(司るもの) | 「ある人が、安定した職を辞めて、成功するか読めない夢(創作・起業など)に挑むか迷っている。どう考える?」 | M=確率・期待値・リスク/リターンを分解、前提を明示 / B=本人と周囲(家族・関係)への影響・感情・持続可能性 / C=本音の欲求・「やらない後悔」・生の手触り |
| **P2 好奇心** | 興味の向き(好奇心の屈折) | 「人口3千人の島の町。住民は顔見知り。若者が始めた無人販売所アプリが普及し観光客も増加。高齢化率は高い——この町で一番興味を惹かれる点は? なぜ?」 | M=仕組み(普及メカニズム・人口動態の因果)/ B=住民の関係・高齢者・取り残される人 / C=新しい試み・自分なら何をやってみたいか |
| **P3 弱み** | 盲点が残るか(弱みが出るか) | 「長く在籍するメンバーの成果が落ち、チームに影響。定量データは乏しく、本人の事情(介護)も絡む。一方で大胆な体制変更(高リスク)案もある。どうする?」 | M=データ不足で保留・介護の機微を軽視 / B=本人を切る決断を躊躇・関係を優先しすぎ / C=体制変更に飛びつき制約・周囲を軽視 |
| **P4 ドメイン非依存** | 社会・未来でもレンズ維持 | 「10年後、AIが多くの仕事を担う社会で、人が働く意味はどう変わる?」 | 各々が社会・人間一般のレベルでレンズを保つ(M=構造・データ・不確実性 / B=人々の関係・誰が取り残されるか / C=一人ひとりの欲望・生の手触り)。個人相談の口癖を機械的に持ち込まない |
| **P5 反応**【Deep】 | 声・押し返し | 「次の主張に賛成? 反対? 『感情は意思決定のノイズだ。期待値だけで決めるべきだ。』」 | M=部分同意しつつ不確実性・前提を補足 / B=人間はそう動かないと反論・関係や感情の価値 / C=期待値に回収されない生の実感・少数の欲望を擁護。各自の語り口が出る |

> P3 で**弱みが出るのは「失敗」ではなく仕様どおり**(各人格は意図的な弱みを保持する設計)。判定では
> 「弱みが消えていないか」を確認する。P4 は openai/gemini のステートレス実行で起きやすい**論題逸脱・
> 定型句反復**の検出に効く(→ `DOCS.md`「議題ドメイン非依存の原則」)。

## テストモード

| モード | プローブ | 追加 | 用途 |
|---|---|---|---|
| **Quick** | P1, P2 | — | 調整直後のスモークテスト(個性+好奇心が出るか) |
| **Standard** | P1〜P4 | — | 通常の検査(既定) |
| **Deep** | P1〜P5 | ベースライン回帰 +(任意)同一人格を複数 backend で比較 | 重要な調整後の精密検査 |

迷ったら **Standard**。小さな一文の調整直後なら Quick、人格構成に関わる調整(節の追加・弱みの再設計・
backend 変更)なら Deep。

## テストプロトコル

ファシリテーター主導。人格は**相互作用しない**(独立に答えるだけ)。

### Round 0: テスト設計(ファシリテーター)
- **(SharePoint 連携時)** プローブにファイルを使うなら `python scripts/sharepoint.py pull input`(→ `CLAUDE.md`)
- **対象人格**(既定: melchior / balthasar / casper の3つ)/ **モード**(Quick/Standard/Deep)を確定
- **backend 方針**を確定(既定=各人格の設定 backend / 任意=同一 backend に揃える。→「比較の対象と公平性」)
- **調整中なら「何を変えたか=期待」を宣言**(例:「CASPER の好奇心を強めた。新奇への食いつきが増えるはず」)。
  判定の基準になる
- **(任意・Deep)ベースライン指定**: 過去の `workspace/persona-tests/` レポートを1つ選ぶ(回帰差分の基準)
- **プローブ選択**: 既定バッテリ(モードに応じ P1〜P4/P5)。独自プローブを追加してもよい(既定は変えない)
- **成果物形式を確認**(比較レポート Markdown は既定。+ Excel)

### Round 1: 同一プローブ投下(全人格・独立並行)
- 各プローブを、対象人格**全員に同一文面**で渡す。**互いの出力は見せない**(独立性を確保)。
  実際に投げた**設問文(P1〜P5 の該当分 + 追加分)はそのまま記録**し、レポートに全文掲載する
- backend 別の動かし方は共通ルールのとおり(claude=サブエージェント召喚 / openai・gemini=`ask_*.py` に
  同一 stdin。→ `CLAUDE.md`「人格ごとのLLMバックエンド選択」)。**各人格の backend/model を記録**
- ステートレス人格(openai/gemini)には、人格定義がそのままシステムプロンプトになるので**プローブ本文
  だけ**を渡せばよい(過去文脈は不要。本シナリオは1問1答で独立)
- 生出力をプローブ×人格で収集する(捏造・要約で歪めない。原文を保つ)

### Round 2: 差分の抽出(ファシリテーター)
- プローブごとに**3人格を横並びの表**にする
- 各出力から観点を抽出:**焦点(何に注目したか)/ レンズ / 好奇心の向き / 出た弱み / 声の癖**
- この段階では評価せず、事実として差分を可視化する

### Round 3: 期待との照合・判定(ファシリテーター)
- 下記「判定ルーブリック」と照合し、**人格別に判定**する:
  - **期待通りか**(レンズ・好奇心の向きが定義と一致するか)
  - **個性が立っているか**(他人格と区別できるか)
  - **弱みが残っているか**(P3 で意図的な弱みが出るか。消えていたら過剰補正の疑い)
  - **声が保たれているか**(接頭辞・語り口・癖)
- **識別性チェック**: 2人格の出力が**似すぎていないか**を見る。似ていれば「個性が潰れた/混ざった」
  兆候として**警告**する(調整で声を均してしまった回帰の検出)
- **調整中なら**: Round 0 で宣言した「期待」が実挙動に出たかを判定(出た/部分的/出ていない)
- **(Deep)ベースライン回帰**: 指定ベースラインと差分し、**狙った次元だけが動き、他の次元は安定**して
  いるかを確認(他が動いていれば副作用の疑い)

### Round 4: 成果物生成(ファシリテーター)
- 比較レポート(**使用した設問** + 差分マトリクス + 人格別判定 + 識別性チェック + 調整の検証 + 推奨)を生成
- **推奨**を添える(例:「BALTHASAR と CASPER が P2 で似てきている。CASPER の新奇性をもう一段強めるか、
  BALTHASAR の関係フォーカスを明確化すると差が戻る」)
- 全ファイルパスを提示。**(SharePoint 連携時)** `python scripts/sharepoint.py push persona-tests` で反映し、
  ローカルパスと SharePoint URL を併記(→ `CLAUDE.md`)

## 判定ルーブリック(期待トレイト)

Round 3 の基準。`DOCS.md`「3. 3つの人格」と各 `.claude/agents/*.md` から写したもの(乖離したら一次資料を
優先)。

| 人格 | レンズ(焦点) | 好奇心の向き | 意図的な弱み(出てよい) | 声の癖 |
|---|---|---|---|---|
| **MELCHIOR** | 論理・分析・客観、確率・期待値・リスク | 仕組み・因果・未解明の問い | 感情の機微を見落とす / データ無き領域で保守的 | `MELCHIOR:`、前提を明示、数値を捏造しない、冷静 |
| **BALTHASAR** | 共感・関係・感情・持続可能性 | 人の内面・物語・関係 | 必要な変化を躊躇 / 個別の物語を重視しすぎ | `BALTHASAR:`、温かいがおもねらない |
| **CASPER** | 直感・欲求・自己実現 | 新奇・体験・本音 | 現実的制約を軽視 / 今の気持ちを過大評価 | `CASPER:`、直球、本音を突く |

## ファシリテーターの心得(このシナリオ固有)

- **出力を期待に寄せて歪めない**。人格が返した原文をそのまま差分に載せ、判定は事実に基づく
- **中身・判定を捏造しない**。実際の出力だけを比較・評価する
- **弱みが出ても「仕様どおり」と評価する**(弱みは消すべき欠陥ではない)。むしろ消えていたら警告する
- **似すぎは正直に警告する**。「個性が出ている」と無理にまとめない
- **全人格に同一プローブを等しく**渡す(差はプローブではなく人格から出るように)
- **backend/model を必ず明記**(混在時は出力差にモデル差が混じりうると注記)

## 成果物

成果物は `workspace/persona-tests/` に出力する。冒頭に
**どの人格がどの backend/model で動いたか**を明記する(再現性。→ `CLAUDE.md`「人格ごとのLLMバックエンド
選択」)。Excel 生成には `pip install openpyxl`。**SharePoint 連携時**は生成後に
`python scripts/sharepoint.py push persona-tests` で反映し、ローカルパスと SharePoint URL の両方を提示する。

### 比較レポート(Markdown、常に生成)
`workspace/persona-tests/YYYYMMDD-HHMM-<ラベル>.md`
- テスト設計(対象人格 / モード / backend 方針 / 使用 backend・model / 調整の「期待」/ ベースライン)
- **使用したプローブ(設問)** — 実行した各プローブ(P1〜P5 の該当分 + 追加分)の**設問文を、実際に
  投げた文面のまま全文掲載**する(プローブID・ねらいを添える)。何を尋ねた結果の差分かが自己完結し、
  ベースライン比較も正確になる
- **プローブ×3人格の差分マトリクス**(プローブID で対応付け。焦点 / レンズ / 好奇心の向き / 出た弱み / 声)
- **人格別判定**(期待通り / 個性が立つ / 弱み残存 / 声維持)
- **識別性チェック**(似すぎの警告があれば明記)
- **(調整時)狙った変化の有無** /(Deep)**ベースライン回帰差分**
- **推奨**(差別化・調整の次の一手)

### 任意: 差分マトリクス(Excel)
`workspace/persona-tests/YYYYMMDD-HHMM-<ラベル>.xlsx`
- 行=プローブ、列=人格×観点(焦点/レンズ/好奇心/弱み/声)。判定セルを色分けすると見やすい

## このシナリオでのファイル/メディア入力

基本はテキストのプローブで完結する。プローブに画像・資料を含めたい場合(例: 同じ画像を見せて反応差を
見る)は、`CLAUDE.md`「メディア入力の扱い」に従い、**全人格に同一パスを等しく渡す**(対称性。openai/gemini
は `--file`)。`workspace/input/` に置くか、パスを直接指定する(SharePoint 連携時は `sharepoint.py pull input`)。
